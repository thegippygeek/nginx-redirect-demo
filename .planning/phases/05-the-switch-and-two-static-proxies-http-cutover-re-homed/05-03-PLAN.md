<objective>
Re-home the flip: re-point `scripts/flip.sh` and the `make reset` canonical-file restore from
the v1 `proxy` container to the `switch`, so the presenter edits one line in
`switch/active-proxy.conf`, reloads the switch, and the identical `curl localhost:9092`
lands on NEW — the same map-flip + reload mechanism as v1's CUT-01/CUT-02, one tier up. Then
reconcile the flip smoke section and bring `make test` green across the HTTP surface, deferring
the switch-SSH sections to Phase 6 honestly. **This is the Slice B demoable milestone:** the
live on-stage flip works from the new switch surface.

**Phase user story (MVP):** **As a** presenter, **I want to** flip the switch selector from
old to new and reload, and watch the same client command now answer NEW with the status page
confirming it, **so that** the on-stage flip action is unchanged from v1 while running on the
blue-green tier.

Purpose: The flip's gate must now probe the switch's real upstreams (proxy-old/proxy-new), not
the backends — the switch parses `upstream old { server proxy-old; }` on every reload
(Pitfall 2). Output: a working switch-sourced flip and a green `make test`.

API-coverage / schema-gate: N/A — shell scripts + smoke suite; no external API, no ORM/schema.
</objective>

<objective>
Author the three nginx tiers that replace v1's single flip-in-place `proxy`: a front
`switch` (the flip surface + evidence writer) and two static single-upstream proxies
(`proxy-old`→`server-old`, `proxy-new`→`server-new`). This plan produces the four config
files and proves each one loads under `nginx -t`. Wiring them into the running rig and
proving HTTP lands on OLD end-to-end is 05-02; the flip is 05-03.

**Phase user story (MVP):** **As a** presenter, **I want to** keep the client hitting
`app.demo.test:9092` exactly as in v1 while traffic now flows through a front switch and a
static proxy tier, **so that** I can later validate the new stack and roll back instantly
without the client ever changing what it connects to.

Purpose: This is a brownfield restructure — every file here is a reshaping of a verified v1
file (see 05-PATTERNS.md), not a new mechanism. The one genuinely new behaviour is a single
added JSON field (`remote`/`$remote_addr`) on the switch's evidence format.
Output: switch/nginx.conf, switch/active-proxy.conf, proxy-old/nginx.conf, proxy-new/nginx.conf.

<assumption_delta_decision>
Pluralization fired: v1's flip selected the active *backend*; v2's switch selects the active
*proxy*. Decision: **promote**. The switch's `map` now selects a proxy, but the variable
stays `$active_backend` and the `old`/`new` labels are retained for continuity with v1's
mechanism and the status page's existing vocabulary. Backend identity still rides the
`X-Backend` response header end-to-end. One-line change of intent, not a redesign.
</assumption_delta_decision>

API-coverage / schema-gate: N/A — no external API/SDK and no ORM/schema files (nginx configs only).
</objective>

<objective>
Wire the three nginx tiers from 05-01 into the running rig: split compose's single `proxy`
service into `switch` + `proxy-old` + `proxy-new` with the health cascade, move the
`app.demo.test` alias to the switch, re-source the evidence from the switch, and re-point the
presenter's `make up`/`logs`/`reload` surface. Add the client-IP (`remote`) render to the
status service, and reconcile the topology smoke sections. **This is the Slice A demoable
milestone:** the same client command as v1 lands on OLD through a two-proxy tier, and the
status page shows the client's real IP.

**Phase user story (MVP):** **As a** presenter, **I want to** run `make up` and see
`curl app.demo.test:9092` land on OLD through the new switch→proxy-old→server-old chain with
the status page proving where traffic went, **so that** the client sees exactly what it saw
in v1 while I now have a blue-green tier underneath.

Purpose: Make the topology real and provable. The health cascade is the load-bearing startup
ordering (nginx aborts parse on an unresolvable upstream — v1 Pitfall 4, now one tier deeper).
Output: a running 6-service rig, evidence re-sourced from the switch with the client's real IP,
and green topology smoke assertions.

API-coverage / schema-gate: N/A — nginx configs + a Python stdlib status service; no external API, no ORM/schema.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| host → published ports | Only loopback-bound ports (9092/9093/9094) cross to the host; the static proxies and container :22 cross nothing |
| switch → static proxy → backend | Internal; the X-Backend identity chain must stay untampered (EV2-02) |
| status → evidence volume | Read-only; the reporting tier provably cannot alter what it reports |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-01 | Tampering | section_proxy integrity assertions | high | mitigate | Assert exactly one X-Backend passes through a static proxy and that backend= is the backend's own header through two hops. **block_on: high — this is where the integrity threat is TESTED before seal.** |
| T-05-04 | Info disclosure | compose published ports | low | mitigate | switch ports 127.0.0.1-bound; proxy-old/proxy-new declare no ports:; container :22 never published (asserted in section_proxy) |
| T-05-05 | Spoofing | client-supplied X-Backend request header | low | accept | $upstream_http_x_backend reads the upstream RESPONSE header, not the request — no filtering needed; adding one would be unnecessary defense |
| T-05-06 | Elevation | status service mounts | low | mitigate | No Docker socket mounted anywhere; status keeps every mount :ro; no depends_on that could take it down with the switch |
</threat_model>

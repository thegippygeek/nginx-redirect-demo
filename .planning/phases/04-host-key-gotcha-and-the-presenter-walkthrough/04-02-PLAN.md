## Phase Goal

**As a** presenter, **I want to** deliberately trigger the SSH host-key failure after the cutover and
fix it live without touching the client, **so that** the audience sees the thing that actually breaks
on migration day and the answer that scales to a real fleet.

**This plan's slice of that goal:** confidence. The presenter can ask the rig whether the gotcha is
armed and whether the fix still works, and get an answer in under twenty seconds instead of finding
out in front of people.

## Locked decisions this plan implements

| Decision | How this plan honours it |
|---|---|
| **D-47** the failure must be reachable on demand, and real | The section causes the failure by regenerating the server's key material and connecting — it never fabricates a trust record or a warning line |
| **D-48 (as corrected)** no new storage for the client's trust record | The section reads and checksums the default location and clears it on the way out. It creates nothing that outlives the client container |
| **D-52** the two modes stay two modes | The section defines its own exported presenter-mode option set and carries a negative control proving test mode is unaffected in the same state. `section_ssh` and `scripts/verify.sh` are not edited |
| **D-58** the client's self-update behaviour is pinned off | This is what makes the trust-record checksum assertion meaningful rather than a false alarm |
| **D-59** the fix is the signal, not the copy | Asserted as presented-fingerprint equality followed by a real connection, never as a file listing |
| **D-51** the fix is reversible | The re-arm is exercised inside the section, so the reversibility claim is a test result rather than an assumption |

## Plan-level decisions (Claude's discretion, exercised here)

- **The section is placed last in the `all` chain**, after `section_ssh`, for the same reason
  `section_ssh` follows `section_cutover`: it is the most destructive, and it leaves the rig in the
  state the next thing expects. Nothing runs after it.
- **The section drives the real scripts, not copies of their commands.** Reimplementing the transfer
  and the re-arm inline would produce a suite that passes while the presenter's targets are broken —
  precisely the class of failure this phase is about.
- **Every assertion label names its requirement ID**, matching the convention every existing section
  uses, so a failure line tells the reader which requirement just regressed.
- **The browser beats are deliberately out of scope here.** Incognito, the redirect contrast and the
  status page were human-verified in Phases 1 and 2; re-automating them in this section would be
  scope creep with no new evidence.

### Rig exclusivity — this plan owns the rig for wave 2

This plan runs in wave 2 alongside Plan 04-03, and the two are file-disjoint. That alone would not make
them safe to run together: this plan drives **one shared, stateful Docker rig**, flipping the selector
to `new`, regenerating `server-new`'s host keys and clearing the client's trust record, restoring only
on the way out. A sibling asserting that routing lands on `old`, or invoking the full suite, would be
reading a selector this plan is actively rewriting — producing not merely flaky failures but results
that **can pass by luck**.

**This plan therefore holds exclusive use of the rig for the whole of wave 2.** Plan 04-03 is a
documentation plan and every check in it is static — no container call, no suite invocation — which is
recorded in its own plan-level decisions and is mechanically visible in its diff (two markdown files,
nothing else). Nothing else may run against the rig while this plan is executing.

**The capture idiom is mandatory and non-negotiable.** Assign every SSH invocation into a variable

invocation on the left of a pipe. Wrap every invocation in an external timeout — the connect timeout
bounds only the TCP connect, not the banner exchange or authentication.

Then the five beats, in order:

1. **Arm and prime.** Run the re-arm script, then flip the selector to `old` and settle it. Connect in
   presenter mode and assert it succeeds and reports the OLD identity. Assert the client's trust
   record now holds an entry for the proxied hostname — the record is keyed on the name the client
   typed, and priming against a backend's own service name records a different, useless entry that
   would make the whole section pass vacuously. Capture the record's checksum into an exported
   variable here.
2. **Flip.** Flip the selector to `new` and settle it with the existing oracle.
3. **The failure (KEY-02).** One assertion checking **both** halves: a non-zero status **and** the
   changed-identification warning text in the captured output. Then the **negative control**: in the
   identical state, the test-mode option set connects cleanly and reports the NEW identity. Without
   the control, a bug that broke SSH outright would still satisfy the first assertion.
4. **The fix (KEY-03).** Before applying it, assert the two backends' presented fingerprints differ —
   that is the precondition, and asserting it here is what makes the post-fix equality meaningful.
   Then run `scripts/fix-hostkeys.sh`. Assert the two backends now present the **same** ed25519
   fingerprint, read from the running rig rather than from a file listing. Add the concurrency edge:
   open a backgrounded, timeout-bounded session that sleeps and then reports its hostname, apply the
   fix while it is running, wait for it, and assert it completed and still named its own backend — the
   daemon re-execs on the signal rather than restarting, so an in-flight session is not dropped. Bound
   and wait for every backgrounded session; a leaked one pins an old worker generation and makes later
   observations non-deterministic across takes.
5. **The success (KEY-04).** The identical presenter-mode invocation now succeeds and its banner names
   the NEW identity. Then the checksum assertion: the client's trust record is byte-identical to the
   value captured in beat 1. This assertion is only satisfiable because presenter mode pins the
   client's self-update behaviour off; note that in the comment, with the measured byte figures, so a
   future maintainer who removes the pin understands what they have broken.
6. **The re-arm, asserted (KEY-01).** Run `scripts/rearm.sh` and assert the presented fingerprints
   differ again and the client's trust record no longer exists. A re-arm that silently did nothing is
   the failure this assertion exists to catch.

Finally, restore: flip back to `old`, settle, clear the traps, and assert the section leaves the rig
selecting `old`, using the same liveness oracle `section_ssh` uses at its own foot.

Add two static guards over the new section, each stripping full-line comments **first** — the
section's own prose names every hazard it guards against, so an unstripped grep would be invalidated
by the very warning that makes it trustworthy — and each asserting the stripped region is non-empty

Scope both guards to this section's own region using the `awk` range idiom the existing guards use,
with the bracketed-letter trick so a guard does not match its own text.

Do not edit `section_ssh`, `scripts/verify.sh`, or any Phase 3 assertion. Their test-mode pins exist
precisely so this phase cannot break them.

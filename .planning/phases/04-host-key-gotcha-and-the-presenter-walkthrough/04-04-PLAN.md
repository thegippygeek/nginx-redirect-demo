<objective>
Make the document unable to rot, and be honest about the one thing that cannot be tested.

Plan 03 wrote `WALKTHROUGH.md`. This plan asserts the four things about it that a machine genuinely
can assert — every command runnable, every path real, no undefined prerequisite, every step
structurally complete — and then stops, and says plainly that passing all four means the document is
self-contained and executable, not that it is comprehensible.

The remaining claim, ROADMAP criterion 5, is that someone who has never seen the demo can follow it
cold. No assertion can make that judgement. It gets an explicit human check and is recorded as
judgement, not as evidence.

Purpose: WALK-01, WALK-02, WALK-03 mechanically; criterion 5 honestly.

Output: `section_walkthrough` in `scripts/smoke.sh`, its wiring, the phase gate, and the cold-read
sign-off.

## Phase Goal

**As a** presenter, **I want to** deliberately trigger the SSH host-key failure after the cutover and
fix it live without touching the client, **so that** the audience sees the thing that actually breaks
on migration day and the answer that scales to a real fleet.

**This plan's slice of that goal:** durability and honesty. The script stays true to the rig, and the
one claim that rests on judgement is labelled as judgement.

## Locked decisions this plan implements

| Decision | How this plan honours it |
|---|---|
| **D-54** every step carries command, expected output and takeaway | Asserted structurally: no step may be missing any of the three, and the three appear in a fixed order |
| **D-55** the fixed narrative order | Asserted as an ordering check over the step headings |
| **D-57** the known traps are documented | Each trap is asserted present by name |
| **04-VALIDATION** criterion 5 gets an explicit human check and is not claimed as mechanically covered | A structured backstop must-have that makes the verifier abstain and escalate, plus a human check block naming the exact protocol |

## Plan-level decisions (Claude's discretion, exercised here)

- **The lint extracts from the document rather than grepping for a hand-maintained list.** A list of
  expected commands maintained beside the document is a second source of truth that drifts from the
  first. Pull the commands and paths out of the document itself and check each against the repository.
- **`section_walkthrough` runs before `section_hostkey` in the `all` chain.** It is non-destructive
  and fast, and the destructive section must stay last for the reason already documented there.
- **The human check is written as a protocol, not a request for approval.** "Read it and say if it
  looks good" produces a yes. "Run it top to bottom on a clean checkout and report every point at
  which you had to guess" produces the information criterion 5 actually needs.

   given.

   command, expected output, takeaway — and that within each step they appear in that fixed order.
   Assert the count of each block type equals the count of step headings, which is what catches a step
   missing one block as well as a stray block belonging to no step. Separately assert the step
   headings appear in the fixed narrative order.

Then assert each trap named in the decisions is present in the document by name: the incognito window
for the cached redirect, the client-container prefix for SSH, the redirect port not following the
flip, `make reset` as the re-arm path, and the verify script's structural blindness to host keys.

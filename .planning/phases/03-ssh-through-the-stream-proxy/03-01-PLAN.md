<objective>
Make SSH into a backend work, non-interactively, with a key the presenter never sees — and make the
backend say who it is the instant the connection opens.

This is the first vertical slice of Phase 3. It is deliberately NOT "the backend layer": it delivers
a complete, presenter-usable capability end to end — from the `client` container, over the Docker
network, into a real sshd, with an identity banner coming back — that simply does not go through the
proxy yet. Plan 02 adds the proxy hop; Plan 03 adds the assertion tool.

Purpose: BACK-04 and BACK-05 are the two requirements every later Phase 3 claim rests on. If the
banner is not `Banner`, SSH-03 and EVID-04 cannot read it. If key auth is not silent and
non-interactive, the presenter gets a password prompt on stage and the verify script blocks forever.

Output: a rendered `/etc/ssh/banner` on each backend, an sshd drop-in that actually takes effect, a
`demo-keys` named volume holding a keypair generated at first run, and a `section_ssh` in the smoke
suite that proves all of it.
</objective>

## Phase Goal

**As a** presenter, **I want to** `ssh` to port 22 on the same host and land on whichever backend is
currently active, **so that** the audience sees the cutover is a real TCP-level migration and not
just an HTTP trick.

**This plan's slice of that goal:** the presenter can SSH into a named backend and it identifies
itself. The proxy hop and the flip are Plan 02.

## Locked decisions this plan implements

| Decision | How this plan honours it |
|---|---|
| **D-41** Key-based auth from the client container, keypair generated at build or first run, public key installed on both backends; password auth stays as a documented fallback | `client/entrypoint.sh` generates the keypair on first run into the `demo-keys` volume; both backends read it `:ro`; `PasswordAuthentication` is never set, so Alpine's `yes` default keeps `demo:demo` alive |
| **D-42** The SSH login banner names the backend as OLD or NEW with its hostname, driven by the same `BACKEND_ID` env var as everything else (D-16) | `backend/templates/banner.template` is rendered by the SAME `envsubst "$VARS"` allowlist already used for `default.conf.template` and `index.html.template` |
| **D-43** The banner must appear before the shell prompt and be visible even for a non-interactive `ssh … <command>` invocation | Implemented as sshd's `Banner`, never as `/etc/motd` + `PrintMotd`. 03-RESEARCH §Q3 measured motd as **absent** for `ssh host command` even with a forced pty — motd is not an alternative here, it is a wrong answer |
| **D-16** One image, two identities | No new env var. The banner joins the HTML body and the `X-Backend` header as a third surface fed by one variable |

## Plan-level decisions (Claude's discretion, exercised here)

- **The banner is ONE line and is byte-identical to that backend's HTTP `/whoami` body.** Two fields,
  identity then hostname, single space, single newline. This is not cosmetic: it means the anchored
  assertions `^OLD server-old$` / `^NEW server-new$` that already prove BACK-03 over HTTP prove
  BACK-04 over SSH unchanged, and the presenter can say the two protocols return the same sentence.
  A multi-line ASCII box was rejected — the banner lands on stderr in every scripted invocation and
  would read as noise in `verify.sh`'s output (Plan 03).
- **Key distribution: `demo-keys` named volume, written by a new client entrypoint.** No committed
  private key (a PEM block in git reads as a real credential leak to anyone who clones or scans the
  repo, however obviously throwaway — unlike `demo:demo`, which is visibly a joke). No second image.
  No host state. No imperative `make up` step, so raw `docker compose up -d --wait` still produces a
  working rig (D-20).
- **`client/Dockerfile` gains an `ENTRYPOINT` only.** `openssh-client`, `openssh-keygen` and
  `/usr/bin/timeout` are already present from Phase 1's D-18 foresight — **verified in the running
  container**. Do not re-add them. `ssh-copy-id` is NOT present; do not plan around it.
- **Every sshd setting this phase adds goes through the drop-in, uniformly.** `Banner` happens to
  work as a plain append (there is no active `Banner` line in Alpine's stock config) but mixing two
  mechanisms invites Pitfall S-2 the next time someone adds a directive.

**The capture idiom, used by every SSH assertion in this file.** Assign the invocation into a shell

accepts the TCP connection while the upstream never replies would otherwise hang on sshd's login
grace period. `timeout` in this image returns 143 on SIGTERM, not 124 — test that the status is
non-zero, never for a specific code. Grepping the captured VARIABLE afterwards is fine; the
prohibition is on piping the invocation itself.

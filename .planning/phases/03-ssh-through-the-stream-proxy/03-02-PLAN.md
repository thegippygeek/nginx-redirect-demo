## Phase Goal

**As a** presenter, **I want to** `ssh` to port 22 on the same host and land on whichever backend is
currently active, **so that** the audience sees the cutover is a real TCP-level migration and not
just an HTTP trick.

**This plan's slice of that goal:** the proxy hop and the flip. Plan 01 made SSH into a named backend
work; this plan makes the presenter stop naming the backend.

## Plan-level decisions (Claude's discretion + the researcher's four open questions, decided)

1. **Stream log field naming — emit BOTH the uppercase label and the raw selector on one line.** The
   demo's whole point is that one word drives both protocols, and reusing the field name the HTTP log
   already uses means the existing `logs-demo` awk colours SSH lines for free. The honesty cost is
   real and is paid explicitly: the HTTP log's field carries the BACKEND asserting its own identity,
   while the stream log's carries the PROXY reporting its own selector — nginx cannot read an
   identity out of an opaque TCP stream. Keeping the raw selector on the same line makes that visible
   to any reader, and a config comment states it outright. **Decided: option (a) plus the comment
   plus the raw selector retained.**
2. **Do not persist the client's `known_hosts` in Phase 3.** The assertions here are deliberately
   orthogonal to host-key state. Whether Phase 4 wants persistence across `docker compose down` is
   entirely Phase 4's decision; flagged in the handover note below so it is not rediscovered.
3. **No `make logs-demo` change.** Contingent on decision 1, and decision 1 removes the need. An
   assertion below pins the coupling so a future rename of the field cannot silently uncolour SSH.
4. **Do not add an SSH proof to `scripts/flip.sh`.** It is a carefully tuned six-step sequence with
   measured timings, and adding a handshake to the money-shot command adds latency and a new live
   failure mode. `verify.sh` (Plan 03) is the right home, and D-44 says so. `scripts/flip.sh` is on
   this plan's do-not-modify list.
5. **Do not build a stream-side analogue of the `$backend_is_valid` guard.** A stream block cannot
   serve a 503 an SSH client could render; an SSH client receiving arbitrary bytes just fails
   differently and less legibly. The proxy's own `[error] no port in upstream` line plus the failure
   status in the stream access log is already a legible diagnosis, and the HTTP guard catches the
   same typo two seconds earlier in the presenter's day. Say this in a config comment so a future
   reader does not think it was overlooked.
6. **Do not extend the compose healthcheck to cover :22.** `guard_check` writes an invalid selector
   on purpose; an end-to-end port-22 probe would flap the container unhealthy mid-suite for exactly
   the reason the healthcheck already avoids :9092. And an SSH handshake every three seconds is not
   a cheap probe. The smoke suite is the right altitude — the plan adds the two assertions there.

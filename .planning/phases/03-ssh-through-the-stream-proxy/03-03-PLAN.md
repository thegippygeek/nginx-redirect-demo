<objective>
One command that answers "did the cutover actually land, on both protocols?" — and that can say no.

This is the last vertical slice of Phase 3. Plan 01 made SSH into a backend work; Plan 02 put the
proxy in the path and made the flip move it. This plan gives the presenter a single command that
issues both an HTTP request and an SSH connection, reports which backend answered each on its own
labelled line, and exits non-zero when the answer is wrong — including the case the demo actually
cares about, where one protocol flipped and the other did not.

Purpose: EVID-04 and EVID-05. The phase's claim is that one word drives two protocols; the only
honest way to close that claim is a tool that would fail loudly if it stopped being true.

Output: `scripts/verify.sh`, a `make verify` target, the Phase 3 presenter section of the README, and
the EVID group of `section_ssh` — which tests the script the same way the script tests the rig.

## Phase Goal

**As a** presenter, **I want to** `ssh` to port 22 on the same host and land on whichever backend is
currently active, **so that** the audience sees the cutover is a real TCP-level migration and not
just an HTTP trick.

**This plan's slice of that goal:** "…with a script that asserts it automatically" — the second half
of the roadmap's Phase 3 goal sentence.

## Locked decisions this plan implements

| Decision | How this plan honours it |
|---|---|
| **D-44** `scripts/verify.sh <expected>` issues an HTTP request and an SSH connection, reports which backend answered each, and exits non-zero on any mismatch, extending the existing POSIX-shell idiom | Same conventions as `smoke.sh` and `flip.sh`: POSIX `sh`, deliberately not `set -e`, `usage()` with exit 2, a documented exit-code vocabulary. Wired as `make verify` alongside `make flip` |
| **D-45** The script asserts both protocols agree with EACH OTHER, not just each against the expectation | A dedicated branch with its own message and its own exit code, exercised live by the smoke suite via a documented test seam |
| **D-43** The banner is visible for a non-interactive invocation because the verify script reads it programmatically | The script reads the banner from the folded-in stderr; that is what makes `Banner` rather than motd the contract |
| **D-37** The `client` container is the canonical SSH source | The SSH half runs through one `docker compose exec -T client` line, from the host, matching `flip.sh`'s execution model |

## Plan-level decisions (Claude's discretion, exercised here)

- **The script runs from the host and shells out for the SSH half.** The HTTP half already needs a
  host-side request against the published port; splitting the script across two execution contexts
  would be worse than a single exec line. This matches `flip.sh` exactly.
- **Exit-code vocabulary: 0 success, 1 mismatch against the expectation, 2 usage, 3 protocols
  disagree with each other.** A distinct code for the disagreement is what lets the smoke suite prove
  D-45's branch is reachable rather than merely present in the source. Document the vocabulary in the
  script's header comment.
- **Two independent readings per protocol.** The SSH banner is the contractual identity claim
  (BACK-04/SSH-03) because it is the backend asserting its own identity; the remote command's stdout
  corroborates that a shell really ran there. Take both. This is D-45's "both must agree" principle
  applied one level down.
- **A documented test seam for the disagreement branch.** The script honours an environment variable
  naming the SSH target host, defaulting to the proxied demo hostname. The smoke suite points it at a
  named backend while the selector says the other one, which produces a genuine, non-simulated
  disagreement and exercises the branch end to end. Document it in the script header as a test seam,
  not as a presenter option.
- **Grep the captured output; never compare it.** With the host-key options in play, ssh emits a
  "permanently added" notice on stderr on every run. That line is expected and harmless — an equality
  comparison against the capture would break on it, which is why every reading is a grep.

**Behavioural assertions.**

- **EVID-04, the happy path.** With the rig selecting OLD, `sh scripts/verify.sh old` exits 0 and its
  stdout contains one labelled line naming the HTTP reading and one naming the SSH reading, each
  carrying the anchored two-field identity of the OLD backend.
- **EVID-04 ordering edge.** In that same output, the line number of the HTTP reading is strictly
  less than the line number of the SSH reading. Use `awk` over a captured copy of the output.
- **EVID-04 adjacency edge.** Point the script's SSH target at an unreachable name so the SSH half
  fails outright, and assert that BOTH labelled lines are still present in the output and the exit
  status is non-zero. A tool that omits the reading it could not take forces the presenter to infer
  which one is missing, on stage.
- **EVID-04 empty edge.** Invoked with no argument, and again with an unrecognised argument, the
  script prints its usage line and exits **2** — a distinct code, so a fumbled invocation can never
  be misread as a failed cutover.
- **EVID-04 concurrency edge.** The whole run completes well inside an external bound even when the
  SSH target is unreachable. Wrap the invocation in a generous `timeout` and assert it was not the
  bound that ended it. This is the "a hung SSH connection blocks the suite" hazard: without the
  script's own internal timeout wrapper, an accepted-but-unanswered connection would wait on sshd's
  login grace period and stall the whole run.
- **EVID-05, mismatch.** With the rig selecting OLD, `sh scripts/verify.sh new` exits non-zero and
  its output names the mismatch. Assert the status is non-zero and that it is the mismatch code
  rather than the usage code — the two must never be confused.
- **EVID-05, D-45 disagreement, exercised live.** With the rig selecting OLD, point the script's SSH
  target at the NEW backend by name. Both readings are then individually valid and they disagree.
  Assert the exit code is the dedicated disagreement code and the message is distinct from the plain
  mismatch message. This is the branch the demo actually cares about, so assert it runs — not merely
  that it exists in the source.
- **EVID-05, the whole rig, end to end.** Flip to NEW, `sh scripts/verify.sh new` exits 0; flip back
  to OLD, `sh scripts/verify.sh old` exits 0. The section's trap restores the selector.

**Static guards (each over the COMMENT-STRIPPED script).**

- **Pitfall S-1.** No line in `scripts/verify.sh` places an ssh invocation on the left of a pipe.
  03-RESEARCH measured a piped invocation returning 0 while the output read that host key
  verification had failed — a pipeline's status is the last command's. This is the single most
  important guard in the phase, and it must strip comments so the script's own warning about the
  hazard cannot invalidate it.
- **Pitfall S-3.** The script contains no quiet or log-level-lowering ssh option. Each of them
  suppresses the pre-auth banner entirely, turning the captured output into the empty string — this
  is the most likely way a future maintainer breaks EVID-04 while cleaning up noisy output. Comments
  stripped, for the same reason.
- **Pitfall S-5.** The script contains no forced-pty option. Unnecessary, since the banner needs no
  pty, and a forced pty with no stdin was measured hanging indefinitely.
- **Pitfall S-7, the Phase 4 guard.** The script DOES contain both host-key options, and a comment

  mentioning the phase — the comment is part of the contract here, because these options are the one
  thing in this repo that could teach an audience a bad habit.
- **Every SSH invocation is externally bounded.** Assert the script wraps its ssh invocation in
  `timeout`. The connect timeout bounds the TCP connect only; it does not cover the banner exchange

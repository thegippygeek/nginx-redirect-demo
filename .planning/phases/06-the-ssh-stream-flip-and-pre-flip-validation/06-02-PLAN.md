<objective>
Deliver the milestone's new payoff: the presenter can prove the new stack is live over BOTH
protocols BEFORE committing to the flip. `curl app-new.demo.test` → NEW and `ssh app-new.demo.test`
→ server-new's banner, both while live traffic on `app.demo.test` still lands on OLD. This is
assertion work, not plumbing — proxy-new already carries HTTP:80 + static SSH:22 + the
`app-new.demo.test` alias since Phase 5. Add a `--target app-new` mode to verify.sh (both probes
run from the client container, since the alias is Docker-DNS-only), a `make verify-new-stack`
surface, and a non-destructive `section_validate` that pins VAL-01/VAL-02/EV2-04. **This is the
Slice B demoable milestone:** validate over both protocols before the cutover.

see the new server answer on both protocols while `app.demo.test` still lands on OLD, **so that** I
commit to the cutover only after I have watched the new stack work.

Purpose: pre-flip validation is what the two-proxy topology buys over v1 — the new stack is
reachable directly through proxy-new's alias, bypassing the switch. Output: a re-usable
`verify.sh --target app-new` mode and a green pre-flip section.

assumption-delta / api-coverage / schema-gate: N/A — shell + POSIX-sh harness; no external API,

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client container → app-new.demo.test | the only path to the new stack pre-flip; resolvable only inside the demo network |
| host → static proxies | the static proxies must expose nothing to the host — app-new is not reachable from `localhost` |
| verify.sh SSH probe → its own exit code | a pipeline could mask a host-key or connection failure as a false PASS |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-06-04 | Info disclosure | static proxies exposing host ports (T-05-04) | medium | mitigate | proxy-old/proxy-new publish NOTHING; app-new is a Docker-DNS alias probed ONLY from the client container. No compose `ports:` entry is added; VAL-01/VAL-02 and `verify.sh --target app-new` run via `docker compose exec -T client` |
| T-06-05 | Tampering | verify.sh / section_validate false PASS via a piped ssh | medium | mitigate | The capture-then-read-`$?` idiom is preserved and asserted region-scoped (no ssh invocation on the left of a pipe); a host-key or connect failure reads as non-zero, never as a silent success |
| T-06-06 | Spoofing | app-new banner is the only identity over SSH | low | accept | The banner is proxy-new's backend's own pre-auth claim relayed opaquely; VAL-02 corroborates it with a `hostname` reading of `server-new`, and no proxy tier synthesises identity |
| T-06-SC | Tampering | supply chain | low | accept | No package installs this phase — no npm/pip/cargo surface; all images pinned since v1 |

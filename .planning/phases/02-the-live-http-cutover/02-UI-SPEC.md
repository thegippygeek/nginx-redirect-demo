# Phase 2 — UI Design Contract

> Visual and interaction contract for the cutover status page (EVID-02, EVID-03; D-23..D-29).
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## What This Surface Is

A **single projected status page**, on screen for roughly 90 seconds, watched by a room during one
event: the nginx upstream flips from OLD to NEW.

| Property | Value | Source |
|----------|-------|--------|
| Viewing distance | Up to ~10 m, 1920×1080 projector, possibly bright room | D-23 |
| Interaction | **None.** No controls, no navigation, no forms, no hover, no focus states | D-24 |
| Refresh | 1000 ms poll, non-interactive | D-24 |
| Lifetime on screen | ~90 s per take, re-run repeatedly | D-36 |
| Job | **Evidence.** The audience must believe traffic moved, and see it happen | EVID-02/03 |

Design consequences that follow and are binding for the whole document:

1. **No interaction states exist.** `:hover`, `:focus`, `:active`, cursor affordances, tooltips,
   transitions-on-input, and disabled styling are all **out of contract**. Anything that looks
   clickable is a defect, because it teaches the audience the presenter is driving the page when
   they are not.
2. **No scroll.** Everything fits 1920×1080 at all times. Content that does not fit is dropped by an
   explicit rule (see `## Overflow & Density`), never by a scrollbar.
3. **Two seconds of attention per element.** Every number on the page is legible without the viewer
   choosing to read it.

## Design System

| Property | Value |
|----------|-------|
| Tool | **none** — hand-written CSS, single stylesheet, no build step |
| Preset | not applicable |
| Component library | **none** — introducing one contradicts the disposability constraint (PROJECT.md) |
| Icon library | **none** — no icon font, no SVG sprite. Glyphs limited to the ASCII/Unicode set below |
| Font | System stack only. **No `@font-face`, no CDN, no network request of any kind** (ENV-03) |

**shadcn gate:** not run. The project is not React/Next.js/Vite; it has no language runtime and no
`package.json`. `components.json` absent, confirmed by scan. Registry safety: not applicable.

**Asset rule (hard, from ENV-03 "runs entirely locally"):** the served HTML must contain **zero**
external references. No `<link>` to a CDN, no Google Fonts, no external JS, no remote images.
Everything is inline `<style>` / inline `<script>` or same-origin static files from the status
container. A demo that shows a broken layout on a conference network with no internet has failed.

**Font stack (verbatim):**

```css
--font-ui:   system-ui, -apple-system, "Segoe UI", Roboto, "Helvetica Neue", Arial, sans-serif;
--font-mono: ui-monospace, SFMono-Regular, Menlo, Consolas, "Liberation Mono", monospace;
```

`--font-ui` matches Phase 1's `system-ui, sans-serif` exactly, so the banner word renders in the
same face as the backend pages. `--font-mono` is a deliberate second family, used **only** in the
recent-requests table for timestamps and paths — tabular alignment across rows is what makes the
OLD→NEW boundary readable as a boundary rather than as noise. It is never used for the banner,
labels, or counters.

**Permitted glyphs beyond A–Z 0–9 and basic punctuation:** `→` (U+2192), `▲` (U+25B2),
`■` (U+25A0), `·` (U+00B7). Nothing else. No emoji — projector colour profiles and font fallback
make emoji unreliable, and they undercut the "this is what production evidence looks like" framing.

## Spacing Scale

Declared values (all multiples of 4). Projection needs the upper end of the scale, so the tokens are
weighted large.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Hairline offsets, chip inner gap |
| sm | 8px | Tight label-to-value gap |
| md | 16px | Table cell vertical padding, inline gaps |
| lg | 24px | Table cell horizontal padding, chip padding |
| xl | 32px | Intra-panel padding |
| 2xl | 48px | **Page safe margin**, gap between panels |
| 3xl | 64px | Gap between the verdict block and the evidence row |
| 4xl | 96px | Vertical padding inside the TRAFFIC banner |

**Exceptions (declared, deliberate, all still multiples of 4):**

- **Projector safe area: 48px on all four sides.** Non-negotiable. Overscan and keystone correction
  eat the outer edge of a projected frame; anything inside 48px is at risk of being cropped or
  trapezoidally distorted. No text, rule, or fill boundary that carries meaning may sit outside it.
- **Flip-boundary rule: 8px tall.** Below `md` because it is a rule, not a spacing value.
- **Table row height: 68px fixed** (not scale-derived). Derived from `32px label type × 1.3 lh + 2 × 16px md`
  rounded to a multiple of 4. Rows must be uniform height so the boundary rule lands at a
  predictable pixel position across takes.

## Layout

Fixed 1920×1080. No media queries, no responsive behaviour, no mobile layout. If the projector is
1280×720 the page scales via a single root `font-size` clamp (below); the grid proportions are
identical.

```
┌────────────────────────────────────────────────────────────────────────┐ 0
│  48px safe margin                                                      │
│  ┌──────────────────────────────────────────────────────────────────┐  │ 48
│  │ CONFIG STRIP                                     height 152px    │  │
│  │  CONFIG SAYS          [ NEW ]                                    │  │
│  └──────────────────────────────────────────────────────────────────┘  │ 200
│                       ↕ 24px + sync marker                             │
│  ┌──────────────────────────────────────────────────────────────────┐  │ 264
│  │ TRAFFIC BANNER                                   height 396px    │  │
│  │  TRAFFIC SHOWS                                                   │  │
│  │           O L D                                                  │  │
│  │  last answered by server-old · 14:02:29                          │  │
│  └──────────────────────────────────────────────────────────────────┘  │ 660
│                       ↕ 64px (3xl)                                     │
│  ┌───────────────────────────────────────┐ ┌─────────────────────────┐ │ 724
│  │ RECENT REQUESTS         1176 × 308    │ │ STATS RAIL  600 × 308   │ │
│  │  8 rows @ 68px, newest first          │ │  OLD  1,204             │ │
│  │  ── FLIP LANDED 14:02:31 ──           │ │  NEW      37            │ │
│  │                                       │ │  SINCE FLIP    0:07     │ │
│  └───────────────────────────────────────┘ └─────────────────────────┘ │ 1032
│  FOOTER  updated 14:02:38 · src /var/log/nginx/access.log · ▲          │ 1032→1080
└────────────────────────────────────────────────────────────────────────┘
```

**Grid:** `display: grid` on the page root; the evidence row is
`grid-template-columns: 1176px 48px 600px` (table, gap, rail). Fixed pixels, not fractions — the
table's column widths must not reflow when a path string grows, or the boundary rule appears to move.

**Root scale hook (verbatim — do not rewrite):**

```css
html { font-size: calc(100vw / 120); }   /* = 16px at 1920px wide */
```

Every size below is expressed in `rem`. One line, no media queries, exact proportional fidelity at
1280×720 and at 3840×2160. The `px` values quoted throughout this document are the 1920-wide
reference values.

**Do not write this as `calc(16px * (100vw / 1920))`.** Dividing by a length is not permitted inside
`calc()` — the divisor must be a plain number. That form is invalid CSS, the whole declaration is
dropped, `font-size` silently falls back to the 16px UA default, and the entire rem scale collapses
to its 1920 reference values regardless of viewport. On a 1920 projector it looks correct, which is
what makes the bug dangerous. `100vw / 120` divides a length by a number and is valid.

**Vertical budget is fixed and there is no scrollbar anywhere on the page.**
`html, body { overflow: hidden; height: 100%; }`

## Typography

Exactly **4 sizes** and exactly **2 weights**. Both limits are hard; a fifth size or a third weight
is a checker failure.

| Role | Size | Weight | Line Height | Letter Spacing | Used For |
|------|------|--------|-------------|----------------|----------|
| Hero | 200px / 12.5rem | 700 | 1.0 | -0.02em | The TRAFFIC SHOWS word (`OLD` / `NEW` / `UNAVAILABLE` / `—`) |
| Display | 72px / 4.5rem | 700 | 1.05 | -0.01em | The CONFIG SAYS word, the two request counters, the since-flip clock |
| Label | 32px / 2rem | 700 | 1.3 | +0.08em (uppercase only) | Section labels, table header, table body, boundary-rule caption |
| Meta | 22px / 1.375rem | 400 | 1.4 | 0 | Footer, hostname sub-line, timestamps outside the table |

**Weights: 400 and 700 only.** No 300, no 500, no 600, no 800. Rationale: `system-ui` resolves to
different families per OS (SF Pro / Segoe UI Variable / Roboto) and intermediate weights either
synthesise or snap unpredictably across those families. 400 and 700 are guaranteed real cuts
everywhere, so the page looks the same on the presenter's Mac and on a borrowed Windows laptop.

**Hero sizing note.** Phase 1's backend banner is `22vw` (≈422px at 1920) because it is the only
thing on a full-viewport page. The status page's hero is 200px — deliberately smaller, because it
shares the frame with the config strip and the evidence row, and because at 200px/700 the word `OLD`
subtends roughly 1.6° of arc at 10 m, comfortably above the ~0.3° legibility floor. It is still the
largest thing on screen by a factor of 2.8× over the next size.

**Uppercase discipline.** Hero, Display, and Label are all rendered uppercase via
`text-transform: uppercase` with the `+0.08em` tracking on Label only (large sizes do not need it and
tracking a 200px word apart weakens the shape). Meta stays sentence case — it is the only text on the
page a viewer is not expected to read from the back of the room.

**Numerals.** Counters and the since-flip clock use `font-variant-numeric: tabular-nums`. Without it
the counters visibly jitter their width on every 1 s poll, which reads as instability in exactly the
element that is supposed to project confidence.

## Color

**Background: dark. Justified, not assumed.**

The page is dominated by two large saturated colour fields (the TRAFFIC banner and the table rows),
and those fields *are* the signal. On a **white** ground, ambient washout on a weak projector lifts
both the ground and the fills, and the fills — which are mid-luminance — converge toward the ground
and lose separation. On a **dark** ground, washout lifts everything uniformly toward grey but the
fills remain the brightest chromatic objects in the frame; hue separation survives even when the
ground has degraded to charcoal. Dark also matches how the Phase 1 backend pages present — a
full-bleed saturated field — so the colour reads as *emitted*, the same way it does on the backend
banner the audience saw five minutes earlier.

Not pure black: `#0b0f14` rather than `#000`. Pure black on a projector is whatever the room's
ambient light is, which makes the "black" unpredictable and makes any 1px hairline vanish. A slight
lift gives the panel edges something to sit against.

| Role | Value | Contrast | Usage |
|------|-------|----------|-------|
| Dominant (60%) | `#0b0f14` | — | Page ground, all panel interiors |
| Secondary (30%) | `#161d26` | 1.6:1 vs ground | Panel fills (table container, stats rail, config strip), table header row |
| Accent — OLD | `#b45309` | **5.02:1** vs white text · **3.83:1** vs ground | See reserved-for list |
| Accent — NEW | `#15803d` | **5.01:1** vs white text · **3.82:1** vs ground | See reserved-for list |
| Foreground | `#ffffff` | **19.2:1** vs ground | Hero, Display, Label text |
| Foreground muted | `#9fb0c0` | **8.65:1** vs ground | Meta text, table header labels, footer |
| Hairline | `#2b3542` | 1.9:1 vs ground | 2px panel borders, table row separators |
| Dead / UNAVAILABLE | `#334155` | **10.4:1** vs white text · 1.85:1 vs ground | UNAVAILABLE banner fill only |

**Panel separation is carried by the border, not the fill.** `#161d26` sits at only **1.6:1** against
`#0b0f14` — adequate on a monitor, but the first thing heavy projector washout destroys. Every panel
(table container, stats rail, config strip, since-flip tile) therefore carries a **2px `#2b3542`
border** and that border, not the fill difference, is what defines its edge. The fill is
reinforcement only. An implementation that omits the borders and relies on `#161d26` alone will look
correct on the developer's laptop and dissolve into a single flat field on the projector, which is
the failure mode this note exists to prevent.

`#b45309` and `#15803d` are lifted verbatim from `compose.yaml` (`BACKEND_COLOR` for `server-old` and
`server-new`) — the same values `backend/templates/index.html.template` paints its full-viewport
banner with. **These two hex values must not be adjusted, tinted, or "improved" for the status
page.** The audience learns the colour language once, on the backend page, and must not have to
relearn it here (D-26, `## Specific Ideas`).

**Accent reserved for — exhaustive list. These colours appear nowhere else:**

1. The TRAFFIC SHOWS banner fill (the dominant reading).
2. The CONFIG SAYS chip fill (the smaller reading).
3. Recent-requests table row fills — one solid fill per row, keyed to the backend that answered.
4. The two request-counter tiles in the stats rail.

Not used for: borders, footer text, the since-flip clock, panel headers, the boundary rule, hover
(which does not exist), links (which do not exist), or decoration of any kind.

**Colour-as-fill rule.** `#b45309` and `#15803d` are only ever used as **background fills carrying
white text**, never as a text or border colour. At 5.0:1 against white they are AA-compliant for
normal text and AAA for large text as fills; as *text* on the dark ground they would fall to roughly
2.7:1 and fail. This single rule keeps every accent usage on the page accessible by construction and
mirrors the Phase 1 pairing exactly.

### The finding that governs everything below: OLD and NEW are isoluminant

Measured relative luminance: `#b45309` = **0.1591**, `#15803d` = **0.1594**. They differ by 0.2%.

**In greyscale — which is what a monochrome projector, a washed-out beam, a phone photo of the
screen, or a deuteranope's red-green channel effectively delivers — amber and green are the same
colour.** Their contrast ratio against each other is 1.00:1.

This is not a defect to fix by changing the hexes (they are locked by Phase 1 and were human-verified
there). It is a constraint that makes the non-colour channels **mandatory rather than nice-to-have**.
Every OLD/NEW distinction on this page carries **at least two** of the following, and never colour
alone:

| Channel | Mechanism |
|---------|-----------|
| **Word** (primary) | The literal string `OLD` or `NEW`, uppercase, weight 700. Continues Phase 1's verified discipline — the word carries the signal, the colour reinforces it |
| **Position** | Config reading always above, traffic reading always below. The stats rail always lists OLD above NEW. Never reordered, never sorted by value |
| **Shape** | Table rows carry a 12px solid left edge bar; OLD rows use a **square** cap (`■`-style, sharp corners), NEW rows use a **triangular** cap (`▲`-style, 45° chamfered top). Distinguishable at 10 m in pure monochrome |
| **Rule** | The flip boundary in the table is an 8px white rule with a text caption — a structural object, not a colour change |

A useful acceptance test for the executor: **screenshot the page, convert to greyscale, and confirm a
viewer can still state which backend is active and where the flip happened.** If they cannot, the
implementation has regressed to colour-only signalling.

## The D-27 Dual Reading — the hardest layout problem in the phase

Two readings that may disagree, shown simultaneously, in front of a room, without confusing anyone.

### Hierarchy: TRAFFIC dominates. Deliberately.

`TRAFFIC SHOWS` is the Hero (200px). `CONFIG SAYS` is Display (72px), 2.8× smaller, in a shorter strip
above it.

Rationale: **config is intent, traffic is evidence.** The presenter already knows what they typed —
they typed it ten seconds ago in front of everyone. What the room does not yet know, and what the
whole phase exists to prove, is where the packets actually went. Evidence outranks intent, and the
element that changes during the money shot must be the biggest thing on screen. Inverting this
(config large, traffic small) would put the audience's eye on the thing that already finished
changing.

### Reading order is enforced by position, not by colour

Config on top, traffic below, always. Config is *cause*, traffic is *effect*, and top-to-bottom is
how the room reads causality. The two are never side by side — side-by-side placement makes them
look like peers to be compared, when the point is that one produces the other.

### Between them: the sync marker — a 40px strip that carries the entire D-27 story

This is the element that makes the disagreement legible. It sits in the 64px gap between the config
strip and the traffic banner, centred, 40px tall.

**State AGREE (config == traffic):**

```
  ────────────────────  IN SYNC  ────────────────────
```
A continuous 4px `#2b3542` rule spanning the full width, broken in the centre for the word `IN SYNC`
in Label 32px, `#9fb0c0`, letter-spaced. The rule is **solid and unbroken** — the two readings are
physically connected. Static. Nothing moves.

**State PENDING (config != traffic):**

```
  ── ── ── ── ── ──  ↓ WAITING FOR RELOAD  ── ── ── ──
```
The rule becomes **dashed** (`repeating-linear-gradient`, 24px dash / 16px gap) — the connection is
visibly broken. The caption reads `↓ WAITING FOR RELOAD` in white, weight 700, at Label size, with a
single `↓` arrow pointing from config down toward traffic. The whole marker pulses opacity
`1.0 → 0.55 → 1.0` over 1600 ms, `ease-in-out`, infinite.

This is the **only** infinite animation on the page, and it is permitted because it is not
decorative: a pulse means "this is unresolved and still in flight", it stops the instant the state
resolves, and its stopping is itself part of the event. The audience learns within one cycle that the
pulsing thing is the thing that has not happened yet.

**Additional PENDING affordance:** the config chip gains a 4px white outline (`outline: 4px solid #fff;
outline-offset: 8px`). White, not coloured — it must not compete with the OLD/NEW hues, and it
survives greyscale.

### Disagreement vs agreement, side by side

| | AGREE | PENDING |
|---|---|---|
| Sync rule | Solid, `#2b3542` | Dashed, white, pulsing |
| Sync caption | `IN SYNC`, muted, 400-weight feel | `↓ WAITING FOR RELOAD`, white, 700 |
| Config chip outline | none | 4px white, 8px offset |
| Motion | none anywhere | one pulse, 1600 ms loop |
| Words | `NEW` / `NEW` | `NEW` / `OLD` |

Note the words differ in both states and that is the fallback signal: even with every other channel
stripped, the page literally reads `CONFIG SAYS NEW` above `TRAFFIC SHOWS OLD`. Nobody has to decode
anything.

### The moment of convergence — how it reads as an event, not a flicker

The hard part. A 1 s poll means the change lands on an arbitrary tick, and an instant swap of a 200px
word between two isoluminant colours can genuinely be missed by someone who blinked.

**One-shot convergence sequence, total 900 ms, fires only on `traffic` value change:**

| t | What happens |
|---|---|
| 0 ms | Sync marker's pulse stops mid-cycle at full opacity. Dashes collapse to solid over 200 ms |
| 0–250 ms | Hero word slides up 40px and fades to 0 (`transform: translateY(-40px); opacity: 0`) |
| 250–500 ms | New word slides in from +40px below to 0, fades to 1. Banner background cross-fades `#b45309 → #15803d` over the same 250 ms |
| 250–900 ms | A 6px white ring on the traffic banner (`box-shadow: 0 0 0 6px #fff`) fades in over 150 ms and out over 500 ms — a single, unmistakable flash framing the element that changed |
| 500 ms | Sync caption swaps to `IN SYNC`, config outline removed |
| immediately | The table's flip-boundary rule is inserted and the since-flip clock resets to `0:00` |

Total: under one second, one direction, one flash. It reads as *"that just happened"*.

**Position on motion — taken, and justified.** Motion is **in**, but rationed to exactly three uses on
the entire page:

1. The convergence sequence above (one-shot, 900 ms, on state change only).
2. The PENDING pulse (loop, but semantic and self-terminating).
3. A 1 px footer heartbeat dot, 1 s cycle, so a frozen page is visibly frozen.

**Nothing else animates. Ever.** No spinners, no skeletons, no counter roll-ups, no row slide-ins on
new log lines, no easing on the poll refresh, no progress bars, no ticker.

The reasoning: this page's credibility is its entire value. A page with ambient motion looks like a
dashboard toy, and a room that thinks it is watching a mockup will not accept it as evidence. But an
*instantaneous* swap of the one element the whole demo is about undersells a moment the presenter has
been building toward for five minutes, and risks a viewer simply missing it. The rule is therefore:
**motion is reserved exclusively for marking state transitions the audience must not miss.** Three
uses, each defensible, each removable without breaking comprehension.

**`prefers-reduced-motion: reduce` fallback — required.** The slide and cross-fade become instant, and
the pulse becomes a static dashed rule. In exchange, the white ring becomes **persistent for 2000 ms**
rather than a fade, so the transition is still explicitly marked. Reduced motion must never mean
"the flip happens with no indication at all".

**Row-count guard.** The convergence sequence fires on a change in the *traffic* reading only. It must
not fire on page load, on the first poll after `make up`, on recovery from UNAVAILABLE, or after an
evidence reset (D-36) — in all four cases the value "changed" but no flip occurred. Guard on
`previousTraffic != null && previousTraffic != current && previousTraffic != 'UNAVAILABLE'`.

## The Recent-Requests Table

D-26 item 2. The audit trail. The boundary must be a **line**, not scattered colour.

**Fixed 8 rows, 68px each, newest first, no scroll, no pagination, no header sort.**

| Column | Width | Font | Content |
|--------|-------|------|---------|
| edge bar | 12px | — | Solid accent fill, square cap for OLD / chamfered cap for NEW |
| time | 200px | mono, Label 32px | `14:02:31` — HH:MM:SS, no date, no milliseconds |
| path | 620px | mono, Label 32px | `/whoami` — truncated at 28ch with `…`, never wrapped |
| status | 120px | mono, Label 32px | `200` |
| backend | 224px | ui, Label 32px, 700 | `OLD` / `NEW`, uppercase, right-aligned |

Row fill is the full-width accent colour at 100% with white text. Row separator: 1px `#2b3542`.
Header row: `#161d26`, Meta 22px, `#9fb0c0`, labels `TIME · PATH · STATUS · BACKEND`.

### The boundary treatment

Where two vertically adjacent rows have different backends, the table inserts a **full-width 8px
solid `#ffffff` rule**, with a centred caption sitting in a 48px band of `#0b0f14`:

```
════════════════════════════════════════════════════
        ▲  FLIP LANDED  ·  14:02:31  ·  OLD → NEW
════════════════════════════════════════════════════
```

Caption: Label 32px, weight 700, white, `+0.08em`, with the `▲` glyph and the literal `OLD → NEW`
direction. Total boundary object height: 64px (8 + 48 + 8).

Why this and not just two colours meeting: an 8px white rule is the highest-contrast object on the
page (19.2:1) and it is a *structural* discontinuity — the table visibly breaks in half. It survives
greyscale, survives washout, survives a 10 m viewing distance, and it is captioned with the exact
timestamp so a technical viewer can cross-reference it against the raw `make logs` tail running in
the adjacent terminal (D-30). The colour change is the corroboration; the rule is the claim.

**The boundary rule steals a row.** With the boundary present the table shows 7 request rows plus the
64px boundary object, not 8. This is stated so nobody "fixes" the apparent inconsistency.

### Boundary pinning — the rule that protects the money shot

Newest-first means that after 8 more requests the boundary scrolls off the bottom and disappears —
during the exact 60 seconds the presenter is talking about it. Unacceptable.

**While a boundary is younger than 60 s, it is pinned at row index 3** (4th visual row from the top):
3 newer rows above it, 4 older rows below. New requests push the newest row in at the top and evict
the oldest row from the bottom, but the boundary stays put. After 60 s the pin releases and the table
reverts to plain newest-8, letting the boundary migrate off naturally.

This is the answer to "what gets sacrificed if the table grows": **older post-flip rows are
sacrificed, and pre-flip history is preserved, for 60 seconds.** The boundary is more valuable than
row 5.

**Fresh-take case — few rows exist yet.** After an evidence reset a presenter may issue only two
requests before flipping, so indices 4–7 cannot be filled and there is nothing above the boundary at
the moment of the flip.

**The pin is a ceiling, not a fixed position: the boundary sits at `min(3, post_flip_row_count)`.**

The index *is* the count of rows above the boundary, and rows above the boundary are by definition
post-flip. So at the instant of the flip there are zero post-flip rows and the boundary sits at index
0 — immediately under the header, with whatever pre-flip rows exist beneath it. Each post-flip request
inserts a row above it, moving it down: 0 → 1 → 2 → 3, where it pins for the remainder of the 60 s.

> **Corrected 2026-07-21.** This rule previously read `min(3, pre_flip_row_count)`, which was wrong in
> two ways: `pre_flip_row_count` is fixed at flip time, so the boundary could never "migrate" as the
> same paragraph required; and it is geometrically impossible — an index of 2 asserts two rows above
> the boundary, but with no post-flip traffic yet there are none to place there. The plan-checker
> accepted the contradictory text; the planner caught it. `post_flip_row_count` is the only reading
> consistent with both the migration behaviour and the layout. The status service computes this
> server-side and the page consumes it verbatim.

**Never render blank filler rows.** The table shrinks to its real content — 3 rows and a boundary is
3 rows and a boundary, not 3 rows plus 4 empty ones. Empty rows on a projector read as "the page is
still loading", which undermines the exact evidence the table exists to provide. The table container
keeps its 308px height with `#161d26` fill; the unused space below the last row is simply panel.

## The Stats Rail

D-26 items 3 and 4. 600px wide, `#161d26` panel, 32px padding, three stacked tiles.

| Tile | Height | Content |
|------|--------|---------|
| OLD counter | 108px | `#b45309` fill, white text. Label `OLD` left, Display 72px tabular count right |
| NEW counter | 108px | `#15803d` fill, white text. Label `NEW` left, Display 72px tabular count right |
| Since flip | 92px | `#0b0f14`, hairline border. Label `SINCE FLIP` left, Display 72px clock right |

**OLD is always the top tile. Never sorted by count.** Position is a signal channel (see the
isoluminance finding) and reordering would destroy it.

Counters format with a thousands separator (`1,204`), max 6 characters before the layout is
considered overflowed — see `## Overflow & Density`.

**Since-flip clock format:** `0:07` under a minute, `1:24` under an hour, `59:59+` capped. Before any
flip in the current session it reads `—` with the label `NO FLIP YET`. It updates on every poll, from
a server-supplied flip timestamp, not from a client-side interval — a client interval keeps counting
while the status service is dead, which is precisely the stale-but-plausible failure D-28 forbids.

## States

Four zero-ish states exist and **all four must be visually distinguishable at 10 m.** Confusing any
two of them is a defect, and the risk here is real: three of them are "the table is empty".

### 1. NORMAL — traffic observed, readings agree

Everything as specified above. Sync marker solid, `IN SYNC`.

### 2. NO TRAFFIC YET — service healthy, log readable, zero requests observed

**This is not an error and must not look like one.** The proxy is fine; nobody has hit it yet.

- Hero reads `—` (em dash), Display-weight, `#9fb0c0`, on `#161d26` — **no accent fill**, because no
  backend has answered and asserting one would be a lie.
- Sub-line: `NO TRAFFIC OBSERVED YET`
- Config strip renders normally with its real value — config is knowable without traffic.
- Sync marker: solid rule, caption `AWAITING FIRST REQUEST` (a third caption, distinct from both
  `IN SYNC` and `WAITING FOR RELOAD`).
- Table: empty-state copy, centred, Label 32px `#9fb0c0`.
- Counters: `0` and `0`, tiles still in accent fill (the tiles are labels, not assertions).

### 3. UNAVAILABLE — proxy down or log unreadable (D-28)

**The most important state on the page.** A page confidently showing `ACTIVE: OLD` while nginx is dead
is worse than one admitting it cannot tell.

- **Traffic banner fill becomes `#334155`** — a dead slate, **not red**. Red is deliberately rejected:
  NEW is green, and red-green is the exact axis ~8% of men cannot resolve. A red failure state next to
  a green success state is the single worst available choice.
- Hero reads `UNAVAILABLE`. At 200px this is 11 characters and will not fit 1824px — **UNAVAILABLE
  renders at Display 72px, not Hero**, and is the one declared exception to the Hero slot. Justified:
  the word is long, and shrinking it while everything else stays put is itself a signal that something
  is structurally wrong.
- **The entire page desaturates.** `filter: saturate(0.15)` on the page root. The accent hues drain out
  of the table, the counters, and the config chip. This is unmistakable at any distance and in any
  colour profile: the page has visibly *lost its colour*, which no other state does.
- **A 12px diagonal hazard border** frames the viewport inside the safe area:
  `repeating-linear-gradient(45deg, #334155 0 24px, #0b0f14 24px 48px)`. Shape channel, survives
  greyscale, and no other state has a border at all.
- **Config strip: also blanked.** If the status service cannot read the log it should not be trusted to
  assert config either. Config reads `—`, sync marker reads `CANNOT DETERMINE`.
- **Table rows are blanked, not frozen** — replaced by the error copy. Leaving the last-known rows on
  screen under a desaturation filter is exactly the stale-but-plausible artefact D-28 forbids.
- Counters read `—`, not their last values.

### 3a. Partial failure — one source readable, the other not

The page has **two** independent inputs: the nginx access log (traffic) and `active-backend.conf`
(config). Either can fail alone. Both directions collapse to the full UNAVAILABLE state above —
there is no partial rendering.

| Failure | Rendering |
|---------|-----------|
| Log unreadable, config readable | **Full UNAVAILABLE.** A service that cannot read the log has lost its evidence source; a config reading alone is intent with nothing to corroborate it, and D-27 exists precisely because intent is not proof |
| Config unreadable, log readable | **Full UNAVAILABLE.** Showing `TRAFFIC SHOWS OLD` with config blank invites the room to read the traffic reading as the answer — collapsing D-27's two readings into one, which D-27 explicitly forbids |
| Both unreadable | Full UNAVAILABLE |

Sync caption in all three cases: `CANNOT DETERMINE`.

Rationale for the deliberately blunt rule: a half-lit page is the most dangerous thing this surface
can render. It looks operational, so nobody questions it, and whichever half is still live gets read
as authoritative. **All-or-nothing is the only rendering that cannot be misread**, and D-28's
standard is "admitting it cannot tell", not "telling as much as it can".

The error detail line names the failing source specifically, so the presenter can diagnose without
guessing: `/etc/nginx/demo/active-backend.conf — permission denied`.

### 4. EVIDENCE CLEARED — post-`make flip-old` reset (D-36)

The between-takes state. Structurally identical to NO TRAFFIC YET, plus a 10-second confirmation.

**The confirmation strip is an overlay, not a layout element.** There are zero spare pixels in the
1080px budget — the footer already occupies 1032→1080 and the page is `overflow: hidden`. The strip
therefore does not participate in the grid at all:

```css
.evidence-cleared {
  position: fixed;
  left: 48px; right: 48px;          /* 1824px, flush with the safe area */
  bottom: 48px;                     /* flush with the bottom safe-area edge (y = 968→1032) */
  height: 64px;
  background: #161d26;
  border: 2px solid #2b3542;
  z-index: 10;
  display: grid; place-items: center;
}
```

- Copy: `EVIDENCE CLEARED · 14:09:02 · READY FOR NEXT TAKE`, Label 32px, weight 700, white.
- **Geometry:** the overlay occupies y = 968→1032 — the bottom 64px of the evidence row
  (724→1032), flush with the bottom edge of the 48px safe area. It does **not** cover the footer
  band (1032→1080), which remains fully visible throughout.
- **What it occludes is harmless by construction.** It covers the bottom 64px of the table and stats
  rail, and State 4 is structurally NO TRAFFIC YET — the table is empty and both counters read `0`,
  so nothing evidential is hidden. After 10 s the overlay is removed. Nothing reflows, because
  nothing ever moved.
- **Declared safe-area exception:** the footer band (1032→1080) sits outside the 48px safe area that
  is non-negotiable everywhere else in this spec. This is deliberate — the footer carries only the
  heartbeat and the log path, both presenter-facing diagnostics rather than room-facing evidence. No
  element carrying meaning for the audience may sit outside the safe area.
- Since-flip clock resets to `—` / `NO FLIP YET`.
- Both counters return to `0`.

Rejected alternative: reclaiming 64px by compressing the `3xl` gap to `2xl` while the strip is
present. That reflows the entire evidence row upward for 10 seconds and then back down — the table
and stats rail visibly jump twice, which is exactly the "looks second-hand" impression D-36 exists to
prevent. An overlay changes nothing underneath it.

Without this strip a presenter mid-reset cannot tell "the reset worked" from "the page is broken",
and D-36 exists specifically so the next take does not start looking second-hand.

### The staleness watchdog — a UI-derived constraint on D-24's mechanism

D-24 leaves meta-refresh vs JS poll to planner discretion. **This spec constrains that choice:**
whatever mechanism is used must reach the UNAVAILABLE state within **5 seconds** of the status service
becoming unreachable.

A bare `<meta http-equiv="refresh">` **cannot satisfy this.** When the server dies, the browser
renders its own connection-error page or, worse, leaves the last successful render on screen
indefinitely — a page confidently displaying `TRAFFIC SHOWS OLD` from 40 seconds ago. That is D-28's
forbidden failure, arriving through the refresh mechanism rather than through the log reader.

**Recommended implementation:** `fetch()` poll at 1000 ms against a JSON endpoint, with a client-side
watchdog that transitions the page to UNAVAILABLE if no successful response has been received in
5000 ms. Roughly 25 lines of inline JS, no framework, no dependency. If the planner selects
meta-refresh instead, it must be paired with an equivalent watchdog and the 5 s bound must still hold.

**Footer heartbeat.** The footer carries `updated 14:02:38 · ▲` where `▲` alternates opacity 1.0/0.35
on each successful poll. It is the smallest element on the page (Meta 22px) and it exists for the
presenter standing near the screen, not for the room: a frozen dot means a frozen page. When the poll
age exceeds 3000 ms the footer text switches to `stale 3s` in white on `#334155` — a pre-warning
before the 5 s UNAVAILABLE transition.

## Overflow & Density

Four elements plus the dual reading on one projected frame. What breaks first, and what gives:

| Pressure | Behaviour | What is sacrificed |
|----------|-----------|--------------------|
| Request table grows past 8 rows | Oldest row evicted from the bottom | Old history. The table is a window, not an archive — `make logs` is the archive (D-30) |
| Boundary would scroll off within 60 s of a flip | Boundary pinned at row index 3 | Post-flip rows 4–7. The boundary outranks them |
| Path longer than 28 characters | Truncated with `…`, never wrapped | Path detail. A wrapped row breaks the 68px uniform height and shifts the boundary rule's pixel position |
| Counter exceeds 6 characters (`999,999`) | Switches to `1.0M` compact notation | Precision. Nobody in the room is reading the 6th digit at 10 m |
| Two flips inside the 8-row window | Only the **most recent** boundary renders | The older boundary. Two white rules in one table reads as a striped table, not as an event |
| Viewport narrower than 1280px | Root `font-size` scales; layout proportions hold | Nothing — but this is out of contract; the target is a 1920×1080 projector |

**Explicitly not provided:** scrollbars, pagination, "show more", column resizing, responsive
stacking, a mobile layout, a print stylesheet. Every one of them is a way for content to hide, and
hidden evidence is not evidence.

## Copywriting Contract

Every user-visible string on the page, verbatim. Voice: **flat, factual, mechanical**. This surface
reports what a log file says; it does not editorialise, celebrate, or apologise. No exclamation
marks, no "Success!", no "Oops", no first person, no emoji. Its authority comes from sounding like
instrumentation.

| Element | Copy |
|---------|------|
| Primary CTA | **None.** This surface is non-interactive by contract (D-24). No buttons, links, or controls exist. A CTA here would be a defect |
| Config strip label | `CONFIG SAYS` |
| Traffic banner label | `TRAFFIC SHOWS` |
| Traffic sub-line | `last answered by {hostname} · {HH:MM:SS}` |
| Sync — agree | `IN SYNC` |
| Sync — pending | `↓ WAITING FOR RELOAD` |
| Sync — no traffic | `AWAITING FIRST REQUEST` |
| Sync — unavailable | `CANNOT DETERMINE` |
| Table header | `TIME` · `PATH` · `STATUS` · `BACKEND` |
| Boundary caption | `▲ FLIP LANDED · {HH:MM:SS} · {FROM} → {TO}` |
| Stats labels | `OLD` · `NEW` · `SINCE FLIP` |
| Since-flip, no flip yet | `NO FLIP YET` |
| Empty state heading | `NO TRAFFIC OBSERVED YET` |
| Empty state body | `Send a request to http://app.demo.test:9092/ and it will appear here.` |
| Error state heading | `UNAVAILABLE` |
| Error state body (log unreadable) | `Cannot read the proxy access log. Not showing a stale reading.` |
| Error state body (config unreadable) | `Cannot read the active-backend config. Not showing a stale reading.` |
| Error state body (both unreadable) | `Cannot read the proxy access log or the active-backend config. Not showing a stale reading.` |
| Error state detail | `{log path} — {reason}` (e.g. `/var/log/nginx/access.log — no such file`) |
| Error state remedy | `Check: docker compose ps proxy` |
| Evidence-cleared confirmation | `EVIDENCE CLEARED · {HH:MM:SS} · READY FOR NEXT TAKE` |
| Footer | `updated {HH:MM:SS} · src {log path} · ▲` |
| Footer, stale | `stale {N}s · src {log path}` |
| Destructive confirmation | **Not applicable.** The only destructive action in this phase is the D-36 evidence reset, and it is invoked from the terminal (`make flip-old`), never from this page. The page's sole responsibility is rendering its *effect* — see the evidence-cleared row above |

**`app.demo.test`, never `app.demo.local`** — D-22 is binding and the empty-state copy above is the
one place in this phase where the hostname is hard-coded into user-visible UI text.

**The error body is load-bearing.** `Not showing a stale reading.` is the sentence that converts a
failure into a credibility gain: the page is publicly declining to guess. Do not soften it to
"Please try again" or "Something went wrong".

## Executor Acceptance Tests

Mechanical checks the implementation must pass. These are the concrete form of the contract.

1. **Greyscale test.** Screenshot the page in NORMAL state, convert to greyscale. A viewer can still
   state which backend is active and locate the flip boundary. *(Non-negotiable — OLD and NEW are
   isoluminant at 0.1591 vs 0.1594.)*
2. **Offline test.** Disconnect the network entirely, hard-reload the page. Layout, fonts, and colours
   are identical. Zero failed requests in the network panel.
3. **Distance test.** View at 10% zoom (simulating ~10 m at 1920px). The Hero word and the boundary
   caption remain readable.
4. **Kill test.** `docker compose stop proxy`. Within 5 s the page is UNAVAILABLE — desaturated,
   hazard-bordered, all values blanked. It never shows a stale `OLD` or `NEW`.
5. **Four-state test.** NORMAL, NO TRAFFIC YET, UNAVAILABLE, and EVIDENCE CLEARED are pairwise
   distinguishable from a screenshot alone, with no caption.
6. **Convergence test.** Run `make flip-new`. The convergence sequence fires exactly once, completes
   under 1 s, and does **not** fire on page load, on recovery from UNAVAILABLE, or after an evidence
   reset.
7. **Boundary-pin test.** Flip, then issue 12 requests within 60 s. The boundary rule is still visible
   at row index 3.
8. **No-scrollbar test.** At 1920×1080, with 8 rows and a boundary present, `document.body.scrollHeight
   === window.innerHeight`.
9. **Reduced-motion test.** With `prefers-reduced-motion: reduce`, the flip still produces a visible
   2000 ms white ring on the traffic banner.
10. **Token audit.** Grep the stylesheet: exactly 4 font sizes, exactly 2 font weights, and
    `#b45309`/`#15803d` appearing only as `background`/`background-color` values, never as `color` or
    `border-color`. Every panel selector carries a `2px solid #2b3542` border.
11. **Root-scale test.** Resize the window to 1280px wide and read
    `getComputedStyle(document.documentElement).fontSize`. It must be `10.666...px`, not `16px`.
    A flat `16px` means the `calc()` was dropped as invalid and the rem scale is dead.
12. **Fresh-take boundary test.** Evidence reset, issue 2 requests, flip. At the moment of the flip
    the boundary renders at row index 0 with the 2 pre-flip rows beneath it and no blank filler rows.
    Each subsequent request moves it down one — index 1, 2, then 3 — where it pins.
13. **Partial-failure test.** Make `active-backend.conf` unreadable while the access log stays
    healthy. The page goes **fully** UNAVAILABLE — it does not render a live traffic reading beside a
    blank config.

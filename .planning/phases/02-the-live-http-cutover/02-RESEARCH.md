## User Constraints (from CONTEXT.md)

### Locked Decisions

**Status page — purpose and shape**

- **D-23:** The status page is **on the projector during the flip**, not a confidence monitor and not an after-the-fact artifact. Design for legibility across a room: large type, minimal chrome, high contrast. The audience watches `ACTIVE: OLD` become `ACTIVE: NEW`.
- **D-24:** It **auto-refreshes every 1–2 seconds**. The flip must land on screen with no interaction from the presenter — reaching for a refresh key mid-flip breaks the moment. Meta-refresh or a small JS poll are both acceptable; server-sent events were considered and rejected as more machinery than a demo needs.
- **D-25:** It is served by a **small separate service** — a fourth container. Rejected alternatives: an nginx location on the proxy (nginx alone cannot render a request history without a helper) and a static file regenerated by a script (goes stale unless something drives it on a timer).

**Status page — content**

- **D-26:** Four elements, all four wanted:
  1. **Big colour-coded ACTIVE banner** (OLD / NEW) reusing the Phase 1 backend banner colours, so the visual language is consistent across the whole demo.
  2. **Recent-requests table** (EVID-03) — last N requests with timestamp, path, and answering backend, colour-coded so the OLD→NEW boundary is a visible line in the table. This is the audit trail that proves the flip.
  3. **Per-backend request counters** — running tallies, making the traffic shift quantitative as well as visual.
  4. **Time since last flip** — reinforces that the change was instant.

- **D-27 (load-bearing):** The page shows **configured intent and observed traffic as two separate readings**: "Config says: NEW / Traffic shows: OLD" until the reload lands. This is deliberately not a single merged status. It makes the reload step a visible event the audience can watch happen, rather than hiding the gap between editing the file and nginx picking it up — which is the most instructive part of the mechanism.

- **D-28:** If the proxy is down or the log is unreadable, the page shows an **explicit UNAVAILABLE state**. It must never render a stale-but-plausible banner. A page confidently showing `ACTIVE: OLD` while nginx is dead is worse than one admitting it cannot tell — particularly with an audience watching.

- **D-29:** The status service reads the nginx access log via a **shared named volume**. Mounting the Docker socket to read container stdout was considered and rejected: it is a real privilege escalation and a poor fit for a demo that might be run on someone else's machine.

**Log presentation**

- **D-30:** Both forms of log evidence ship: a **raw tailing terminal** (`make logs`) and the status page's table. The terminal carries authenticity for a technical audience; the table is readable by everyone. The raw log corroborates the pretty view.
- **D-31:** Phase 1's realistic access-log format stays the **default** — it is already working and asserted. A **stripped, demo-legible variant** is added for the `make logs` view (time, path, colour-coded backend), readable from the back of a room. Two views, one underlying log. The main format is NOT simplified — keeping it realistic is what supports the "this is what you'd actually see in production" claim.
- **D-32:** `make logs` covers the **proxy and both backends**. Seeing the request arrive at the backend proves it truly landed there rather than being served from somewhere upstream. Extra noise is an acceptable price for that proof.

**The flip**

- **D-33:** Both command shapes ship: **`make flip`** toggles whatever is currently active (fast), and **`make flip-old` / `make flip-new`** name the target explicitly (unfumbleable when you have lost your place on stage).
- **D-34:** The command **rewrites the one word, prints the before/after diff, and reloads**. The printed diff is what the audience sees change. Editing the file live in an editor is also documented as the more dramatic option for presenters who want it — the `$backend_is_valid` guard from Phase 1 exists precisely to make a live typo fail loudly rather than 502.
- **D-35:** `make flip-new` **refuses to flip when the target backend is unhealthy**, with a clear message. On stage, flipping to a dead box produces a 503 the presenter then has to explain; in a real migration this is exactly the pre-cutover check you would want. Small cost, removes a live failure mode.

**Rehearsal between takes (CUT-05)**

- **D-36:** Re-running the demo is **`make flip-old` plus clearing the evidence** — the status page's counters and recent-request history reset so the next take starts clean. Containers keep running; no teardown. A config-only flip-back alone was rejected because carried-over counters and a stale request table would confuse a fresh audience. Phase 1's full `make reset` remains available but is not the between-takes path — using it would undercut the "no teardown needed" point CUT-05 exists to make.

### Claude's Discretion

- Language, base image, and framework for the status service (D-25). Keep it small; the project has no language runtime yet and adding a heavy one contradicts the disposability constraint.
- The status page's port number, consistent with Phase 1's 9090/9091/9092/9093 narration mnemonic.
- Exactly how many requests "recent" means in the table, and how the OLD→NEW boundary is rendered.
- The mechanism for the auto-refresh (meta refresh vs JS poll), subject to D-24.
- How the demo-legible log variant is produced (a second log_format plus a filtering target, or post-processing in `make logs`).
- How "clear the evidence" (D-36) is implemented — whether the status service exposes a reset endpoint, truncates its own state, or the log is rotated.
- Whether the status service polls the log or tails it.

### Deferred Ideas (OUT OF SCOPE)

- **Making the 9093 redirect follow the flip** — would require an explicit decision; today its staticness is a feature. Not this phase.
- **SSH cutover (CUT-04)** — Phase 3. The flip mechanism built here must be extensible to the `stream` context, but Phase 2 does not implement it.
- **Weighted / gradual traffic shifting** — explicitly out of scope for the whole project; the demo is a single decisive cutover.
- **Server-sent events for the status page** — considered under D-24 and rejected as excess machinery. Revisit only if polling proves visibly laggy on a projector.
- **A merged single-status reading** — deliberately rejected by D-27; do not "simplify" the two readings into one.

## Summary

Phase 2 is mechanically small — one nginx directive, one new container, four Makefile targets — but it contains three failure modes that would each cost a rework cycle and one that would fail live on stage. All four were reproduced and characterised in this session.

**The flip is decisive, but not atomic.** Measured across five runs against the live Phase 1 stack with a tight request loop: in three of five runs, one to three requests issued in a **26–90 ms window after `nginx -s reload` returned** were still served by the old upstream, interleaved with new-upstream responses, before settling permanently. Separately, an established HTTP keep-alive connection — the browser's situation — was **closed by nginx** 250 ms after the reload and landed on NEW on reconnect, so there is no stale-connection risk for the audience. `worker_shutdown_timeout` is not needed here, and D-24's rejection of SSE happens to be what keeps that true. The practical consequence: `make flip` must settle-and-verify rather than assert immediately, and CUT-03 should be claimed as "decisive within 100 ms", which is honest and still overwhelming on stage.

**The log-sharing question (Q3) resolves cleanly, and the obvious implementation is a trap.** nginx permits multiple `access_log` directives in one context, so `access_log /dev/stdout demo;` can stay byte-identical — Phase 1's `docker compose logs proxy | grep -q 'backend=OLD'` assertion is provably safe — while a second directive writes a purpose-built `escape=json` format to a shared volume. The trap is the mount point: mounting a named volume at `/var/log/nginx` pre-populates it from the image layer, symlinks included, so nginx follows `access.log -> /dev/stdout`, no real file is ever created, and the sidecar's `cat` on that symlink **hangs indefinitely rather than erroring**. On stage that is a frozen status page, not an UNAVAILABLE banner. Mount at `/var/log/demo` instead.

**The health gate (D-35) is more load-bearing than the decision text implies.** With `server-new` stopped, `nginx -t` and `nginx -s reload` both fail outright with `host not found in upstream "server-new:80"` — because `upstream new { ... }` is parsed unconditionally — and the master **silently keeps running the previous config**. The file said `new`, exit code was 1, and traffic kept going to OLD. That is precisely D-27's PENDING state, stuck forever. So the gate must probe **both** backends, not just the target, and `make flip` must check the reload's exit status.

**Primary recommendation:** add a second `access_log` writing `escape=json` to a named volume at `/var/log/demo`; add an unpublished nginx listener on 8081 carrying `access_log off;` that serves both the container healthcheck and a `$active_backend` reload oracle; and implement the status service as a **stateless, stdlib-only Python 3.13-alpine container with no Dockerfile** that recomputes everything from the log file and `active-backend.conf` on each request. A working prototype of that service was built and exercised through all four UI-SPEC states in this session.

## Architectural Responsibility Map

| Capability | Primary Tier | Secondary Tier | Rationale |
|------------|-------------|----------------|-----------|
| Selecting the active upstream | **Proxy (nginx)** | — | `proxy/active-backend.conf` is the single source of intent (D-12). Nothing else may hold a copy |
| Recording who answered each request | **Proxy (nginx)** | — | `$upstream_http_x_backend` is the backend's own assertion (D-11). No other tier may synthesise it |
| Producing machine-readable evidence | **Proxy (nginx)** | Shared volume | A second `access_log` with `escape=json`. Parsing an ad-hoc text format in the sidecar would be the tier error |
| Deriving traffic reading, counters, boundary | **Status service** | — | Pure function of the evidence log. Deriving it in nginx is impossible; deriving it in the Makefile would go stale |
| Reading configured intent | **Status service** | — | Reads `active-backend.conf` directly. D-27 requires the two readings be *independently* sourced — the status service must never learn config from the log or vice versa |
| Rendering the projected page | **Browser (client-side)** | Status service | Inline JS polls a JSON endpoint (UI-SPEC's 5 s watchdog requirement). Server-side templating cannot detect its own server dying |
| Proxy liveness | **Status service** probe | Compose healthcheck | A readable log file is not proof nginx is alive. Needs an active probe against the proxy (Pitfall 6) |
| Executing the flip | **Makefile** | Proxy | Rewrite, diff, gate, reload, verify. The status service is strictly read-only and never mutates config |
| Clearing the evidence (D-36) | **Makefile → proxy container** | Shared volume | The status volume is mounted `:ro` in the status service by design; truncation is issued into the proxy container |

**Tier boundary that matters most:** the status service is **read-only in every direction** — `:ro` on both mounts, no Docker socket, no write path. It cannot flip, cannot reset, and cannot corrupt the evidence it reports. That is what lets it be believed.

### Pattern 1: Dual `access_log` — one nginx, two audiences

**What:** nginx permits multiple `access_log` directives in the same context; every one of them receives every request. `[VERIFIED: isolated Compose project, 3 requests → 3 identical lines on stdout and 3 JSON lines in the volume file]`

**When to use:** whenever a second consumer needs the log in a different shape and the first consumer's format is already asserted against — exactly Phase 1's situation.

```nginx
# proxy/nginx.conf — inside http { }

# UNCHANGED from Phase 1. Do not add fields, do not reorder, do not add a
# timestamp. Phase 1 asserts `docker compose logs proxy | grep -q 'backend=OLD'`
# and D-31 requires this format stay realistic.
log_format demo '$remote_addr -> $host:$server_port "$request" '
                '$status upstream=$upstream_addr backend=$upstream_http_x_backend '
                'rt=$request_time urt=$upstream_response_time';

# NEW. Machine-readable, one JSON object per line. escape=json is not cosmetic:
# it is what makes a request path containing %0a unable to forge a log line.
log_format evidence escape=json
    '{"t":"$time_iso8601","ms":"$msec","path":"$uri","req":"$request_uri",'
    '"status":$status,"backend":"$upstream_http_x_backend",'
    '"upstream":"$upstream_addr","host":"$host","port":"$server_port"}';

access_log /dev/stdout              demo;       # make logs / Phase 1 assertions
access_log /var/log/demo/access.log evidence;   # the status service
error_log  /dev/stderr notice;
```

**Why `$msec` as well as `$time_iso8601`:** `$time_iso8601` has one-second resolution, which cannot order requests within the flip window. `$msec` gives epoch-with-milliseconds — used for the since-flip clock and for ordering. Note the existing `demo` format carries **no timestamp at all**; the status page's `HH:MM:SS` column has no source without this second format. `[VERIFIED: raw stdout lines inspected — no time field present]`

### Pattern 2: The stateless status service

**What:** every HTTP request to `/api/status` recomputes the entire world from two files. The service holds no counters, no cursor, no cache, no history.

**Why this is the right shape here:**

| Property | Consequence |
|----------|-------------|
| No stored state | D-36's evidence reset is a single `: > access.log`, and counters, table, boundary and since-flip clock all reset together, atomically |
| No file offset | Immune to truncation-under-a-reader, the classic tailing bug |
| No write access | Volume mounts `:ro` in both directions; the service that reports the evidence provably cannot alter it |
| Restart-safe | `docker compose restart status` loses nothing |
| Trivially testable | `build()` is a pure function of two files — smoke assertions can drive it by writing fixture lines |

**Cost:** re-reads and re-parses the whole file each poll. At ~165 bytes/line measured, 10,000 requests is 1.6 MB; a 90-second take produces tens of lines. Not a concern. If it ever were, seek to `max(0, size - 262144)` and let the existing skip-unparseable-line rule absorb the torn first line.

### Pattern 3: Parsing `active-backend.conf` (D-27's intent reading)

**What:** strip comments per line, then match a single anchored pattern.

```python
DEFAULT_RE = re.compile(r"^\s*default\s+([A-Za-z0-9_.-]+)\s*;")

for line in raw.splitlines():
    line = line.split("#", 1)[0]        # comments FIRST — this is load-bearing
    m = DEFAULT_RE.match(line)
    if m:
        return m.group(1).upper(), None
return None, f"{CONF} — no `default <backend>;` line found"
```

**Why comments must be stripped first:** the canonical file's second line is
``# Change `old` to `new` to cut over. Nothing else.`` — it contains both backend
words. Any parser that greps the whole file for `old`/`new`, or that matches
`default` without anchoring to line start, will read the presenter's comment as
config. `[VERIFIED: parser returned "OLD" against the real byte-identical file]`

**Also handle:** a `map` block may legitimately gain non-`default` entries (a per-port override). The parser should ignore them but the status service should surface their presence, because Phase 3 (D-13) will `include` this same file from the `stream` block and the map is keyed on `$server_port`.

**Definition of "traffic shows X" — already decided by UI-SPEC, do not re-open.** The traffic sub-line copy is `last answered by {hostname} · {HH:MM:SS}`, which is literally the most recent logged request. Use **last request**, not a window or a majority. A debounce would delay the money shot by one request; a time window would leave the reading ambiguous when the presenter pauses to talk. The ~90 ms interleave (Finding 1) can in principle produce a one-poll flicker, but only for requests issued *concurrently with* the reload — a presenter typing a curl by hand is never inside that window, and UI-SPEC's "only the most recent boundary renders" rule already absorbs the extra transitions.

**Rows that must be excluded from the traffic reading, the counters, and the table:**

| Case | Log signature | Why excluded |
|------|---------------|--------------|
| The 9093 static redirect | `"port":"9093"`, `"backend":""` | Deliberately does not follow the flip (Phase 1 known constraint). Counting it would misreport the cutover |
| The `$backend_is_valid` 503 guard | `"status":503`, `"backend":""` | nginx answered directly; no backend was consulted |
| Any request nginx answered itself | `"backend":""` | The table's edge bar is keyed to a backend; there is none to key to |

Filter: `port == "9092" and backend != ""`. Note nginx renders an unset variable as **empty string** under `escape=json`, not as `-`. `[VERIFIED: JSON output inspected]`

### Pattern 4: The `make flip` pipeline — gate, rewrite, test, reload, prove

Six steps, in this order. Steps 1, 3, 4 and 5 each exist because of a measured failure.

```make
# GNU Make 3.81. Literal $ doubled. No .ONESHELL.
flip-new:
	@sh scripts/flip.sh new

flip-old:
	@sh scripts/flip.sh old

flip:
	@sh scripts/flip.sh toggle
```

The recipe body belongs in a POSIX `sh` script, not in Make: the sequence needs a
retry loop and early exits, and Make 3.81 without `.ONESHELL:` runs each recipe
line in a separate shell.

```sh
# 1. HEALTH GATE — probe BOTH backends, not just the target (Pitfall 3).
for b in server-old server-new; do
  docker compose exec -T proxy curl -fsS --max-time 2 "http://$b/healthz" >/dev/null 2>&1 || {
    echo "REFUSING TO FLIP: $b is not answering /healthz."
    echo "  nginx parses BOTH upstream blocks, so a reload would fail and the"
    echo "  running config would silently stay put. Bring it up:  docker compose up -d $b"
    exit 1; }
done

# 2. REWRITE THE ONE WORD + PRINT THE DIFF (D-34).
cp proxy/active-backend.conf "$tmp"
sed "s/default [a-z]*;/default $target;/" "$tmp" > proxy/active-backend.conf
diff -u "$tmp" proxy/active-backend.conf   # non-zero exit is expected; ignore it

# 3. VALIDATE. Catches a hand-typo AND a backend that vanished since step 1.
docker compose exec -T proxy nginx -t || { cp "$tmp" proxy/active-backend.conf; exit 1; }

# 4. RELOAD — CHECK THE EXIT CODE (Pitfall 4).
docker compose exec -T proxy nginx -s reload || {
  echo "RELOAD FAILED — nginx is still running the PREVIOUS config."
  cp "$tmp" proxy/active-backend.conf; exit 1; }

# 5. PROVE THE RELOAD LANDED, using the running config's own view (Pattern 7).
#    Not a request through :9092 — that would pollute the evidence log.
i=0
while [ "$i" -lt 25 ]; do
  got=$(docker compose exec -T proxy curl -sS --max-time 1 http://localhost:8081/active-backend | tr -d '\r\n')
  [ "$got" = "$target" ] && break
  i=$((i+1)); sleep 0.2
done
[ "$got" = "$target" ] || { echo "reload did not take within 5s (config reports '$got')"; exit 1; }

# 6. ONE REAL REQUEST — settles the interleave window (Finding 1) and seeds the
#    status page's traffic reading so the projector converges immediately.
sleep 0.2
curl -sS http://localhost:9092/whoami
```

**Step 6's `sleep 0.2` is not superstition.** Measured interleave windows were 26 ms, 58 ms and 28 ms after the reload command returned; 200 ms clears all observations with a 3× margin, and step 5 has already consumed real time.

**D-36's evidence reset** attaches to `flip-old` only:

```sh
docker compose exec -T proxy sh -c ': > /var/log/demo/access.log'
```

Issued into the **proxy** container because the status service's mount is `:ro` by design.

### Pattern 5: Evidence reset by truncation

**What:** `: > file`. nginx opens access logs with `O_APPEND`, so after truncation the next write lands at offset 0 with **no sparse NUL hole**. `[VERIFIED: 480 bytes → truncate → 0 bytes → 3 requests → exactly 480 bytes, `od -c` shows the file begins with `{` and not NULs]`

**What NOT to do:** `rm` the file. nginx holds the descriptor and keeps writing to the now-unlinked inode until `nginx -s reopen`. The status service would correctly report `no such file` and the page would go UNAVAILABLE while nginx cheerfully logged into the void. `[VERIFIED indirectly: `mv`ing the file produced exactly that UNAVAILABLE state]`

**Volume lifecycle — a gap the planner must close.** The named volume survives `docker compose down` and is only removed by `down -v`. So:

| Command | Evidence log afterwards |
|---------|-------------------------|
| `make reset` (already `down -v`) | Gone — clean, correct |
| `make down` then `make up` | **Previous take's log survives**, counters resume mid-count |
| `make flip-old` (D-36) | Truncated — clean |

Recommendation: truncate on `make up` as well, or have `up` print the row count it inherited. Silently resuming a prior take's counters is exactly the "looks second-hand" failure D-36 exists to prevent.

### Pattern 6: Two log views, one log (D-30/D-31/D-32)

```make
# D-30/D-32: raw, all three services, authentic. UNCHANGED format.
logs:
	docker compose logs -f proxy server-old server-new

# D-31: the demo-legible view. Timestamps come from Docker (-t), so
# log_format demo is not touched at all.
logs-demo:
	docker compose logs -f -t proxy server-old server-new | awk '\
	  /backend=NEW/ { printf "\033[1;97;42m NEW \033[0m %s\n", $$0; next } \
	  /backend=OLD/ { printf "\033[1;97;43m OLD \033[0m %s\n", $$0; next } \
	  { print }'
```

**Verified:** ANSI escapes emitted by `awk` survive the pipe byte-for-byte (`od -c` shows `033 [ 1 ; 9 7 ; 4 3 m` reaching the terminal). `docker compose logs -t` prepends `2026-07-21T07:33:34.518047594Z`. `[VERIFIED]`

**Match on the `backend=` token by regex, never by field position.** Field indices shift by 1 when `-t` is added and by 2 more when the `proxy-1  | ` service prefix is present — and D-32 requires that prefix so the audience can tell the proxy's line from the backend's line. Regex matching is invariant to both. Note also the `$$0` — Make consumes a single `$`.

**Rejected:** a second `log_format` carrying literal ANSI escape bytes. It would put unprintable characters into a config file the presenter displays on screen, and colour is a terminal concern, not a log concern.

### Pattern 7: The unpublished health/oracle listener

```nginx
# Unpublished — no compose `ports:` entry. Reachable only from inside the
# container and from other containers on the demo network.
server {
    listen 8081;
    access_log off;          # LOAD-BEARING — suppresses BOTH access_log sinks
    default_type text/plain;

    location = /nginx-health   { return 200 "ok\n"; }
    location = /active-backend { return 200 "$active_backend\n"; }
}
```

Serves three jobs at once:

1. **Compose healthcheck for the proxy** (the Phase 1 verifier's flagged gap):
   `test: ["CMD","curl","-fsS","http://localhost:8081/nginx-health"]` — `curl` is present in `nginx:1.30-alpine` `[VERIFIED: /usr/bin/curl]`, so this matches the backends' existing idiom exactly.
2. **The reload oracle.** `return 200 "$active_backend\n"` reports the *running* config's selector, so `make flip` can prove the reload took without touching a backend and without writing an evidence-log line. `[VERIFIED: returned `old` against the real map]`
3. **The status service's independent proxy-liveness probe** (Pitfall 6).

**`access_log off;` at server level suppresses every inherited `access_log` directive** — verified: 10 probes produced zero lines on stdout *and* zero lines in the evidence file. Without it, a 3-second healthcheck interval would push a synthetic row into the projected table every 3 seconds and inflate both counters. `[VERIFIED]`

**Do not healthcheck `:9092`.** Two reasons: `smoke.sh`'s `guard_check` deliberately writes an invalid selector, which the `$backend_is_valid` guard turns into a 503 — the container would flap unhealthy mid-suite; and the probe would itself land in the evidence log.

**`status` must NOT declare `depends_on: proxy: {condition: service_healthy}`.** UI-SPEC acceptance test 4 is `docker compose stop proxy` → the page shows UNAVAILABLE within 5 s. A health dependency would risk Compose taking the status service down with the proxy, leaving nothing to render the state D-28 exists to guarantee. The status service is deliberately decoupled: it reads a volume and probes over the network, and it is *most* valuable when the proxy is dead.

**Port 9094** continues Phase 1's 9090/9091/9092/9093 narration mnemonic. `-u` on `python3` is required or stdout buffering will swallow the service's own diagnostics in `docker compose logs`.

### Anti-Patterns to Avoid

- **Mounting the shared volume at `/var/log/nginx`.** The image ships symlinks there; the result is a hang, not an error. See Pitfall 1.
- **Replacing `access_log /dev/stdout demo;` instead of adding beside it.** Silently breaks two Phase 1 assertions and `make logs`.
- **Adding fields or a timestamp to `log_format demo`.** D-31 locks it. The timestamp belongs in the `evidence` format and, for the terminal view, in `docker compose logs -t`.
- **Health-gating only the flip target.** nginx parses both `upstream` blocks; either backend being down blocks the flip in *both* directions. See Pitfall 3.
- **Assuming `nginx -s reload` succeeded.** It returns non-zero and keeps the old config. See Pitfall 4.
- **Verifying the flip with a request through `:9092` inside `make flip` and nothing else.** Fine as the final confirmation (step 6) but it must come *after* the `:8081` oracle poll — otherwise you cannot distinguish "reload failed" from "reload landed, request raced the window".
- **Rendering log fields with `innerHTML`.** The request path is attacker-controlled and reaches the projected page verbatim. See Pitfall 8.
- **`docker compose restart proxy` as a flip mechanism.** Contradicts D-14 and discards the zero-downtime point entirely.
- **Grepping `docker compose ps --format '{{.Ports}}'` for a literal port.** Phase 1 was already bitten; the proxy still renders `127.0.0.1:9092-9093->9092-9093/tcp`. Use `docker compose port <svc> <port>`.
- **Giving the status service write access to the volume or the Docker socket.** D-29 rejected the socket; the `:ro` mount is what makes the evidence trustworthy.

## Don't Hand-Roll

| Problem | Don't build | Use instead | Why |
|---------|-------------|-------------|-----|
| Machine-readable access log | A custom delimited format + a field splitter | `log_format ... escape=json` + `json.loads` | Handles quoting, backslashes and control characters correctly, and makes `%0a` in a request path unable to forge a log line — verified (Pitfall 8) |
| Incremental log tailing | Offset tracking, inotify, a rotation watcher | Re-read the whole file per poll | The truncation-under-a-reader bug does not exist if you have no cursor. Demo-scale files make it free |
| Counters and since-flip clock | In-memory state in the status service | Derive from the log on every request | D-36's reset becomes one `: >` and all four readings reset atomically and consistently |
| "Did the reload take?" | Parsing `nginx -s reload` stderr, or diffing worker PIDs | `return 200 "$active_backend\n"` on the internal listener | The running config reporting its own selector is the only source that cannot lie, and it costs no evidence-log line |
| Backend health check | Parsing `docker compose ps --format json` | `docker compose exec -T proxy curl -fsS http://<backend>/healthz` | Exercises the exact path nginx will use (Docker DNS + reachability + app), needs no JSON parsing, and is immune to Compose output-format drift |
| Colourised log view | A second nginx `log_format` with embedded escape bytes | `docker compose logs -t \| awk` | Keeps unprintable characters out of a config file the presenter shows on screen |
| HTTP server for the status page | A framework, or a hand-rolled socket loop | `http.server.BaseHTTPRequestHandler` + `ThreadingTCPServer` | Correct HTTP/1.1 keep-alive, Content-Length and connection handling for free; the whole service is ~150 lines |
| HTML-escaping log fields | A custom escaper | `element.textContent = value` in the inline JS | Cannot be got wrong, and UI-SPEC already mandates a JSON endpoint + client-side render |
| Timestamps in the terminal view | Adding `$time_local` to `log_format demo` | `docker compose logs -t` | Honours D-31's "the main format is NOT simplified" without touching a Phase-1-asserted string |

**Key insight:** every one of these is a place where a small piece of custom state would be introduced. In a demo whose entire value is *credibility*, state is the enemy — every cached counter, stored offset and remembered last-value is a way for the projected page to be confidently wrong. The architecture that wins is the one where nothing is remembered and everything is re-derived.

## Common Pitfalls

### Pitfall 1: Mounting the shared volume at `/var/log/nginx` — the highest-risk mistake in the phase

**What goes wrong:** the status service hangs. Not errors — **hangs**. On stage that is a frozen status page, which is worse than a broken one because D-28's UNAVAILABLE state never renders.

**Why it happens:** `nginx:1.30-alpine` ships `/var/log/nginx/access.log` as a symlink to `/dev/stdout` and `error.log` → `/dev/stderr`. `[VERIFIED: ls -la in the running proxy]` When Docker mounts an *empty* named volume over a directory, it pre-populates the volume from the image layer — **symlinks included**. nginx then opens `access.log`, follows the symlink, and writes to stdout; no real file is ever created. The sidecar mounts the same volume, sees `access.log -> /dev/stdout`, and `cat` on it opens *the sidecar's own stdout pipe* and blocks forever.

**Reproduced this session** in an isolated Compose project: the volume contained only the two symlinks, and the `cat` in the sidecar had to be killed after 120 s. `[VERIFIED]`

**How to avoid:** write the evidence log to `/var/log/demo/access.log` and mount the volume at `/var/log/demo`. The path has no symlinks in either image. Never mount at `/var/log/nginx`.

**Warning signs:** the status service's first request never returns; `docker compose exec status ls -la /var/log/demo` shows a symlink rather than a regular file; the evidence file's size stays at 0 while `docker compose logs proxy` shows traffic.

---

### Pitfall 2: Replacing the stdout `access_log` instead of adding a second one

**What goes wrong:** two Phase 1 smoke assertions fail —

```
EVID-01 precondition: proxy log carries backend=OLD
HTTP-02 evidence: proxy log records app.demo.test:9092
```

— and `make logs` goes silent, which is D-30's raw-tail half of the evidence story.

**Why it happens:** it reads like a migration ("move the log to a file so the sidecar can read it") when it is actually a fan-out.

**How to avoid:** keep `access_log /dev/stdout demo;` byte-identical and add a second directive. Verified: three requests produced three identical `backend=` lines on stdout and three JSON lines in the volume file, simultaneously. `[VERIFIED]`

**Warning signs:** `sh scripts/smoke.sh proxy` drops below 17 passes.

---

### Pitfall 3: A stopped backend breaks the flip in *both* directions

**What goes wrong:** with `server-new` stopped, this is what happens:

```
$ docker compose exec -T proxy nginx -t
nginx: [emerg] host not found in upstream "server-new:80" in /etc/nginx/nginx.conf:28
nginx: configuration file test failed          (exit 1)

$ docker compose exec -T proxy nginx -s reload
nginx: [emerg] host not found in upstream "server-new:80" in /etc/nginx/nginx.conf:28
                                               (exit 1)

$ curl -sS http://localhost:9092/whoami
OLD server-old                                 ← still OLD, config file says `new`
```

`[VERIFIED: all three outputs reproduced verbatim this session]`

**Why it happens:** nginx resolves `upstream ... server <name>` at **config parse time**. Both `upstream old` and `upstream new` are parsed on every reload regardless of which one `$active_backend` selects. Docker's embedded DNS drops a stopped container's record, so the parse aborts — and when a reload's config fails to load, the master logs the error and **continues running the previous configuration**. This is exactly the Phase 1 pitfall that motivated `depends_on: service_healthy`, resurfacing at reload time.

**Consequence for D-35:** the health gate must probe **both** backends, not just the target. Flipping *back* to `old` is equally blocked when `new` is down.

**Consequence for the status page:** this produces a permanent D-27 PENDING state — config says NEW, traffic shows OLD, pulsing forever. That is honest and arguably instructive, but the presenter needs the terminal to explain why, which is what the gate's message does.

**How to avoid:** step 1 of Pattern 4, plus checking the reload exit code (Pitfall 4).

**Warning signs:** the sync marker pulses `WAITING FOR RELOAD` and never resolves; `docker compose logs proxy` shows `[emerg] host not found in upstream`.

---

### Pitfall 4: Not checking `nginx -s reload`'s exit code

**What goes wrong:** `make flip` prints a satisfying diff, appears to succeed, and traffic does not move. On stage this is the presenter staring at OLD while saying "and now it's new".

**Why it happens:** `nginx -s reload` writes `[emerg]` to stderr and exits 1, but if the Makefile recipe line is not the last in its rule, or the error is swallowed by a pipe, Make does not stop. Phase 1's `make reload` already has the right discipline (`nginx -t` first, then reload, then a verifying curl) — inherit it rather than writing a new shape.

**How to avoid:** Pattern 4 steps 3–5, and restore the previous file contents on failure so the repo does not end up claiming an intent that never took effect.

**Warning signs:** exit code 1 from the reload; the `:8081/active-backend` oracle still reporting the old value after 5 s.

---

### Pitfall 5: Treating the flip as instantaneous

**What goes wrong:** a smoke assertion or a `make flip` verification issued immediately after reload intermittently sees OLD, and the test is flaky in the worst possible way — green on the developer's machine, red in front of an audience.

**Measured behaviour**, five runs against the live stack with a tight request loop and microsecond timestamps:

| Run | Transitions observed | Interleave window |
|-----|----------------------|-------------------|
| 1 | OLD → NEW @ idx77, → OLD @ idx78, → NEW @ idx81 | ~69 ms, 3 requests |
| 2 | OLD → NEW @ idx78 | none |
| 3 | OLD → NEW @ idx72, → OLD @ idx74, → NEW @ idx75 | ~58 ms, 1 request |
| 4 | OLD → NEW @ idx76 | none |
| 5 | OLD → NEW @ idx78, → OLD @ idx79, → NEW @ idx80 | ~28 ms, 1 request |

3 of 5 runs interleaved; window 26–90 ms; never more than 3 requests; **never any OLD beyond ~100 ms after the reload command returned.** `[VERIFIED]`

**Why it happens:** `nginx -s reload` signals the master and returns. The master spawns new workers and tells the old ones to shut down gracefully; for a few tens of milliseconds both generations are draining the same listening socket, so a connection can still be accepted by an old worker holding the old config.

**The keep-alive case is clean, and this matters more.** An established `Connection: keep-alive` connection — what the projected browser holds — was **closed by nginx** rather than served stale: the next request 250 ms after the reload raised `RemoteDisconnected`, the client reconnected, and got NEW. Every subsequent request on that connection was NEW. Only one worker remained in `ps` afterwards. `[VERIFIED: Python http.client persistent-connection test]`

**`worker_shutdown_timeout` is NOT needed.** Default is unset (old workers linger until their connections close), but this demo has only short HTTP requests. Flag for Phase 3 and beyond: a long-lived connection through the proxy — SSE, a websocket, or the Phase 3 `stream` SSH session — *would* pin an old worker indefinitely. D-24's rejection of SSE is what keeps this true today; Phase 3 should revisit it for the `stream` block, where a live SSH session legitimately outlives a reload.

**How to avoid:** `make flip` polls the `:8081` oracle then waits 200 ms before its confirming request. Smoke assertions settle before asserting. Claim "decisive within 100 ms", not "atomic" — it is still an imperceptible instant on stage and it has the merit of being true.

---

### Pitfall 6: A readable log is not proof the proxy is alive

**What goes wrong:** the presenter runs `docker compose stop proxy`. The evidence log file still exists and is still perfectly readable. A status service that derives its state only from that file keeps rendering `TRAFFIC SHOWS OLD` indefinitely — **exactly the stale-but-plausible banner D-28 forbids**, and a direct failure of UI-SPEC acceptance test 4.

**Why it happens:** D-29's volume-based design decouples the status service from the proxy's lifecycle, which is the point — but decoupling means the file outlives the process.

**How to avoid:** the status service needs an **active liveness probe** in addition to reading the file. Probe `http://proxy:8081/nginx-health` over the Docker network with a short timeout on each poll. It costs no evidence-log line (`access_log off;`) and it fails within the timeout when the container is stopped. Feed the result into the UNAVAILABLE decision alongside the two file reads.

**The full UNAVAILABLE predicate is therefore three-input, not two:** `log unreadable OR config unreadable OR proxy not answering` → UNAVAILABLE. UI-SPEC's `## States 3a` table covers the first two and mandates all-or-nothing; the third input joins them on the same terms.

**Warning signs:** `docker compose stop proxy` and the page keeps showing a backend. That is the single most damaging defect this phase can ship.

---

### Pitfall 7: Healthcheck traffic flooding the projected evidence

**What goes wrong:** a proxy healthcheck pointed at `:9092`, or a status service that probes liveness by requesting `:9092`, injects a synthetic row into the recent-requests table every 3 seconds and inflates both counters. The audience sees requests nobody made, the table's 8-row window fills with noise within 24 seconds, and the flip boundary scrolls away.

**Why it happens:** `:9092` is the evidence path. Anything that touches it is evidence.

**How to avoid:** the dedicated `:8081` listener with `access_log off;`. **Verified:** ten probes against `:8081` produced zero lines on stdout *and* zero lines in the evidence file; a real request to the traffic port produced exactly one line in each. A bare TCP connect (`nc -z`) also produced no line. `[VERIFIED]`

**Warning signs:** the table shows `/healthz` or `/nginx-health` rows; counters climb with no one issuing requests.

---

### Pitfall 8: The request path is attacker-controlled and reaches the projected page

**What goes wrong:** the request path is logged verbatim and rendered in the status page's table. A request to `http://app.demo.test:9092/<script>alert(1)</script>` puts executable markup on the projector if the table is rendered with `innerHTML`.

**Verified this session:**

```
{"path":"/b<script>alert(1)</script>","req":"/b%3Cscript%3Ealert(1)%3C/script%3E","status":200}
```

The payload survives `$uri` intact and correctly — it is a valid JSON string. The escaping burden is entirely on the renderer. `[VERIFIED]`

**How to avoid:** render every log-derived cell with `element.textContent = value`, never `innerHTML`. This also satisfies UI-SPEC's 28-character truncation cleanly (truncate the string, assign as text) with no risk of severing an entity mid-sequence.

**The good news — log injection is already mitigated.** A path containing `%0a` becomes a literal `\n` **inside** the JSON string rather than a real newline:

```
{"path":"/a\n{\"path\":\"FORGED\"}","req":"/a%0a...","status":200}
```

Three crafted requests produced exactly three log lines. A plain delimited format would have been forgeable — meaning an audience member on the presenter's laptop could inject a fake `backend=NEW` row into the projected evidence. `escape=json` is a security control here, not a convenience. `[VERIFIED]`

### Pitfall 9: Compose port-range collapse (carried forward from Phase 1)

**What goes wrong:** a literal grep of `docker compose ps --format '{{.Ports}}'` for `127.0.0.1:9092` fails, because Compose collapses adjacent published ports on one service into a range. Still true today: the proxy renders `127.0.0.1:9092-9093->9092-9093/tcp`. `[VERIFIED]`

**Phase 2's exposure is low but not zero.** The status service is its own service publishing a single port, so no collapse can occur there. The risk is a new assertion written in the old style.

**How to avoid:** the rule Phase 1 already established — `docker compose port <service> <port>`, which resolves one container port at a time and is immune:

```sh
assert "T-02-xx status published on loopback only" \
    'docker compose port status 9094 | grep -q "^127.0.0.1:9094$"'
```

Also note `docker compose ps <svc> --format '{{.Health}}'` returns the **empty string** for a stopped container (and the container is absent entirely without `-a`), not `unhealthy`. An assertion written as `!= "unhealthy"` passes for a dead container. `[VERIFIED]`

import re

DEFAULT_RE = re.compile(r"^\s*default\s+([A-Za-z0-9_.-]+)\s*;")

def read_config(path):
    """Returns (value, error). Comments are stripped BEFORE matching: the
    canonical file's second line is
        # Change `old` to `new` to cut over. Nothing else.
    and contains both backend words. A whole-file grep reads the presenter's
    prose as configuration."""

| Old approach | Current approach | When changed | Impact here |
|--------------|------------------|--------------|-------------|
| Logs to files inside the container, read by a log agent | Logs to stdout, collected by the runtime | Docker-era, ~2014 | The official nginx image symlinks its logs to `/dev/stdout`. Phase 2 needs *both*, which is why the dual `access_log` (and Pitfall 1) exist |

| Meta-refresh dashboards | `fetch` poll + a client-side staleness watchdog | Ongoing | UI-SPEC mandates it: meta-refresh fails to a frozen last-good render, which is exactly D-28's forbidden state |
| `version:` key in `compose.yaml` | Omitted | Compose v2 | Already correct in this repo |

- **`docker-compose` (hyphenated, v1):** gone. This machine runs Compose v5.1.4 as a Docker CLI plugin.

- **Assuming `docker compose ps --format json` field names are stable:** they have churned across major Compose versions. Compose 5.1.4 emits `"Health":"healthy"`, but the health gate deliberately does not depend on it.

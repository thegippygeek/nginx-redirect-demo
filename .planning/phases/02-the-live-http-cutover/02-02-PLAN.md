<api_contract>
## `GET /api/status` — the interface 02-03 renders

`Content-Type: application/json`, `Cache-Control: no-store`, GET only, no CORS headers, no
state-changing verbs.

| Key | Type | Meaning |
|-----|------|---------|
| `state` | `"OK"` \| `"NO_TRAFFIC"` \| `"UNAVAILABLE"` | The rendering mode. `EVIDENCE_CLEARED` is not a server state — it is `NO_TRAFFIC` plus a client-side 10 s confirmation |
| `detail` | string \| null | On `UNAVAILABLE` only: `"{path} — {reason}"`, e.g. the config path followed by a lowercased OS error string. No tracebacks, no environment |
| `failing_source` | `"log"` \| `"config"` \| `"proxy"` \| `"log+config"` \| null | Which input failed, so the page can name it without guessing |
| `config` | `"OLD"` \| `"NEW"` \| other uppercased token \| null | Configured INTENT. Read from `active-backend.conf` only. `null` under `UNAVAILABLE` |
| `traffic` | `"OLD"` \| `"NEW"` \| null | Observed EVIDENCE — the backend of the LAST served row. `null` under `NO_TRAFFIC` and `UNAVAILABLE` |
| `traffic_host` | string \| null | The backend's own `X-Backend-Host` value from the last served row. Empty string if the backend did not send one — never synthesised from the backend word |
| `traffic_at` | `"HH:MM:SS"` \| null | Timestamp of the last served row |
| `sync` | `"IN_SYNC"` \| `"PENDING"` \| `"AWAITING_FIRST_REQUEST"` \| `"CANNOT_DETERMINE"` | Drives UI-SPEC's sync-marker caption. `PENDING` iff `config != traffic` and both are known |
| `counts` | `{"OLD": int, "NEW": int}` | Served-row tallies. Always both keys, always present |
| `rows` | array, newest first, length 0-8 | Each: `{"time":"HH:MM:SS","path":str,"status":int,"backend":"OLD"\|"NEW","ms":float,"bhost":str}`. Already windowed and pinned server-side |
| `boundary` | object \| null | The MOST RECENT transition only: `{"at":"HH:MM:SS","ms":float,"from":"OLD","to":"NEW","row_index":int,"age_s":float}` |
| `since_flip_s` | number \| null | Seconds since `boundary.ms`, computed server-side each poll. `null` when no boundary exists |
| `extra_map_entries` | array of strings | Non-`default` entries found in the map block (Open Question 4) |
| `served_at` | `"HH:MM:SS"` | When this response was computed — drives the footer's `updated` reading |
| `log_path` | string | The evidence log path — drives the footer's `src` reading |

`boundary.row_index` = the number of rows rendered ABOVE the boundary object =
`min(3, post_flip_row_count)`. See `must_haves.flagged_assumptions` for the UI-SPEC ambiguity this
resolves.
</api_contract>

    that opens the file mid-write can still see a partial trailing line and must never raise.

    `probe_proxy()` performs a short-timeout GET against the proxy's internal health endpoint over
    the Docker network and returns a boolean. This is the third input and it is not optional: a
    readable log file is not proof nginx is alive. After a `docker compose stop proxy` the evidence
    file remains perfectly readable, and a service deriving state only from that file keeps
    rendering a confident backend reading indefinitely — the stale-but-plausible banner D-28
    forbids. The probe costs no evidence line because the target listener suppresses its logging.

    OR the proxy is not answering, return `UNAVAILABLE` with `detail` and `failing_source` set and

    nobody questions it, and whichever half is still live gets read as authoritative. Then filter

| Boundary | Description |
|----------|-------------|
| HTTP client -> proxy :9092 -> evidence log -> status service | An attacker-controlled request path travels verbatim from the wire into the JSON the projected page renders |
| status container -> shared volume `demo-logs` | Read-only in this direction by design; the reporter cannot alter what it reports |
| status container -> proxy :8081 over the Docker network | The active liveness probe; the only input that can detect a dead proxy |
| host loopback -> status :9094 | Unauthenticated by design, loopback-bound, exposing only request paths and backend names already visible in `make logs` |

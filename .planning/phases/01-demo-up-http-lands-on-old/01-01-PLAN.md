## Phase Goal

**As a** presenter, **I want to** bring the whole demo rig up with one command and show two visibly distinct, self-identifying backend servers, **so that** the audience understands "here are two separate boxes" before nginx enters the story.

> **Story provenance:** derived by re-shaping the existing ROADMAP.md Phase 1 `**Goal:**` line into canonical user-story form. No new intent was invented. See the planner's return note — the ROADMAP goal line is not itself in `As a / I want to / so that` form.

<objective>
Establish the walking skeleton's foundation and the first demoable beat: one `docker compose up` brings `server-old`, `server-new` and the `client` container up healthy, and the presenter can browse to two adjacent host ports and see a full-screen colour-coded OLD and NEW banner. Both backends are the SAME image differing only by a `BACKEND_ID` environment variable (D-16), each running nginx and sshd under one supervisor (D-17, D-18).

This plan also lays down the test harness (`scripts/smoke.sh`) and the presenter command surface (`Makefile`) that plans 01-02 and 01-03 extend rather than replace.

Purpose: prove "two identical boxes, different identities" — the claim that makes the Phase 2 cutover meaningful. If the boxes are not provably identical, "the only thing that changed is which server answered" is not a claim the demo can honestly make.
Output: `.gitignore`, `Makefile`, `scripts/smoke.sh`, the complete `backend/` image, `client/Dockerfile`, and a `compose.yaml` with three services.
</objective>

<assumption_delta_decision>
**Detector fired:** `pluralization` (a second backend server is introduced where the narrative starts with one) and `chosen` (backend identity moves from a baked constant to a runtime parameter).

**Primary noun after generalization:** `backend` — a parameterised instance of one `demo-backend` image, identified at runtime by `BACKEND_ID`.

**Decision:** `promote`.

**Rationale:** D-16 already promotes the general representation. There is no privileged "old server image" with a "new server" bolted alongside it; both `server-old` and `server-new` are instances of the same image, and the specific identity (`OLD` / `NEW`) is demoted to a per-instance parameter. A third backend, or Phase 4's re-keyed variant of `server-new`, needs no new image and no Dockerfile change — only another compose service with a different `BACKEND_ID`.

**Accepted debt:** none. `add-alongside` was not taken.

**Suggested invariant test (accepted, implemented in Task 1):** `docker compose images` reports the SAME image ID for `server-old` and `server-new` while `/whoami` returns distinct bodies. This goes red the instant a future phase reintroduces a per-backend image, which is exactly the regression that would silently break the "provably identical boxes" claim.
</assumption_delta_decision>

Install `openssh`, `supervisor` and `curl` in one `apk add --no-cache` layer, create an unprivileged `demo` user with a deliberately-obvious throwaway password (RESEARCH § Security Domain, ASVS V2 — this credential is never published to a host port in Phase 1). Build `backend/entrypoint.sh` as `#!/bin/sh` with `set -e`. The explicit envsubst variable allowlist is load-bearing, not cosmetic: without it envsubst consumes nginx's own `$host`, `$remote_addr` and `$hostname` config variables.

Build `backend/supervisord.conf` with `nodaemon=true` under `[supervisord]`, a `[program:nginx]` running nginx with the `daemon off` global directive, and a `[program:sshd]` running sshd in the foreground with `-e`. Both programs need `autorestart=true`, stdout and stderr routed to `/dev/stdout` and `/dev/stderr`, and BOTH `stdout_logfile_maxbytes` and `stderr_logfile_maxbytes` set to `0`. All three of those settings are mandatory — RESEARCH Pitfall 5 verified each one independently causes a startup failure or restart loop.

Build `backend/templates/default.conf.template` as a single `server` block listening on 80 as `default_server`. It sets `add_header X-Backend "${BACKEND_ID}" always;` (envsubst-rendered) and `add_header X-Backend-Host "$hostname" always;` (an nginx variable, left untouched by the allowlist). Root at `/usr/share/nginx/html`. Add `location = /healthz` returning 200 `text/plain` body `ok` with `access_log off`, and `location = /whoami` returning 200 `text/plain` with the body being `${BACKEND_ID}` then a single space then `$hostname` then a newline — this exact two-field order is what the anchored smoke assertions depend on.

Build `backend/templates/index.html.template` per RESEARCH Pattern 3: a full-viewport flex-centred page with `background:${BACKEND_COLOR}`, an `h1` at `22vw` containing the literal `${BACKEND_ID}` text, and a subordinate line showing `hostname: ${BACKEND_HOSTNAME}`. The word itself must carry the signal — the colour is a reinforcement, not the message.

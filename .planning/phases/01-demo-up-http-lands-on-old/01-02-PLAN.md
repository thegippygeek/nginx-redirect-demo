## Phase Goal

**As a** presenter, **I want to** send a request to `app.demo.local:9092` and have nginx transparently land it on `server-old` with the URL unchanged, **so that** the audience sees the migration endpoint working before I flip it.

<objective>
Introduce nginx into the story. The proxy service joins the same `docker compose up`, listens on 9092, and forwards to whichever backend `proxy/active-backend.conf` selects — which in Phase 1 is always `old`. The client container resolves `app.demo.local` through Docker DNS and reaches the proxy directly; the host browser reaches the same name through the D-03 `/etc/hosts` entry.

This plan also settles three things Phase 1 owns on behalf of later phases: the shape of the flip include (D-12 + D-13), the access log format that Phase 2's EVID-01 consumes, and the `$backend_is_valid` guard that turns a typo'd flip into a legible 503 instead of a mid-stage 502.

Purpose: this is the walking skeleton's spine — one real HTTP request traversing nginx, landing on an identified backend, observable in the log.
Output: `proxy/nginx.conf`, `proxy/active-backend.conf`, an extended `compose.yaml` and `scripts/smoke.sh`, and a `make reload` target.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Host loopback -> published port 9092 | Untrusted local requests enter the proxy |
| Client container -> proxy container (Docker DNS `app.demo.local`) | In-network request path |
| Proxy -> backend containers (`proxy_pass`) | The proxy forwards attacker-influenced headers to the backends |
| Presenter's editor -> bind-mounted `proxy/` -> live nginx config | A host-side file edit changes running proxy behaviour on reload |

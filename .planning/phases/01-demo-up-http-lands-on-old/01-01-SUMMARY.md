## Accomplishments

- **One command, whole rig.** `docker compose up -d --wait` comes up clean from cold: `server-old` and `server-new` both reach Docker health status `healthy`, `client` is up, and no cloud account, registry login or credential is involved anywhere.
- **Two provably-identical boxes with distinct identities.** Both backends resolve to a single image ID (`97b09a54…`) yet return `OLD server-old` and `NEW server-new` from `/whoami`. Only `server-old` carries a `build:` key, so the shared-image invariant is structural — a future phase cannot quietly reintroduce a second image without the smoke assertion going red.
- **Both identity signals shipped together (D-10 + D-11).** The literal word OLD/NEW carries the message in the page, the `/whoami` body and the `X-Backend` header independently of the background colour, so a colour-blind viewer or a monochrome projector still gets the signal.
- **Two processes, one container, one supervisor (D-17).** `ps` inside `server-old` shows docker-init as PID 1, supervisord under it, and an nginx master plus an sshd listener side by side — with `autorestart=true` on both so either returns to service without a compose restart.
- **Phase 4's premise preserved.** `ssh-keygen -A` runs in the entrypoint at container start, so the two backends carry different ED25519 fingerprints (`SHA256:DJQ2wb1v…` vs `SHA256:HKdUetHd…`) with container hostnames as key comments — not `root@buildkitsandbox`.
- **The test idiom and the presenter surface are laid down.** `scripts/smoke.sh` dispatches by section with 13 green `backends` assertions and explicit RED stubs naming plans 01-02 and 01-03; the Makefile's target vocabulary is set so `make flip` lands as an extension rather than a second idiom.

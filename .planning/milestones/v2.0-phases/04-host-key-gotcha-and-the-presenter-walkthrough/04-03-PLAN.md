## Phase Goal

**As a** presenter, **I want to** deliberately trigger the SSH host-key failure after the cutover and
fix it live without touching the client, **so that** the audience sees the thing that actually breaks
on migration day and the answer that scales to a real fleet.

**This plan's slice of that goal:** "…and run the entire demo from a written script." The second half
of the ROADMAP Phase 4 goal sentence.

## Plan-level decisions (Claude's discretion, exercised here)

- **The wrong fix is written as demonstration-then-contrast, not as condemnation.** Show the client-side
  deletion succeeding, with no editorial. Then state the two verified facts — it took two steps rather
  than one, and the second was trusting whatever answered next without checking — and ask the room how
  many laptops, CI runners, jump boxes and automation controllers hold that same record. Let the
  audience draw the conclusion. This framing survives either presentational preference and needs no
  arbitration before execution. *(Researcher open question 1, resolved here.)*
- **`make reset` is the documented headline re-arm and the fast in-place path is the footnote.** This
  honours D-51 verbatim while giving a presenter doing three takes in a row the option they will
  actually want, with both measured timings stated so the choice is informed rather than guessed.
  *(Researcher open question 2, resolved here.)*
- **The README's Phase 3 SSH example is corrected in this phase.** It documents an outcome a fresh rig
  does not produce, and it is exactly the class of defect ROADMAP criterion 5 exists to catch. Routing
  it through the presenter-mode target fixes the inaccuracy and introduces the two modes in one edit.
  *(Researcher open question 3, resolved here.)*
- **The transferred public keys keeping their origin host's comment field is narrated, not hidden.** It
  is cosmetic and has no protocol effect, and it is the single most vivid piece of evidence in the
  whole demo that the new server is wearing the old server's cryptographic identity. The walkthrough
  points at it deliberately.
- **The fingerprint in the failure output is shown as varying.** It is the new server's and differs on
  every run; printing a literal would make a correct run look wrong to a careful reader.

### Rig exclusivity — why every check in this plan is static

This plan runs in **wave 2 alongside Plan 04-02**, and file disjointness is not sufficient grounds for
that: both would otherwise verify against **one shared, stateful Docker rig**. Plan 04-02's
`section_hostkey` is the most destructive section in the suite — it flips the selector to `new`,
regenerates `server-new`'s host keys and clears the client's trust record, restoring only on the way
out. Anything running concurrently that asserts routing lands on `old`, or that invokes the full suite,
would be reading a selector another process is actively rewriting. That does not merely produce flaky
failures; it can **pass by luck**, which is worse.

**Plan 04-02 owns the rig for wave 2. This plan touches it not at all.** Every check here is static:
the document exists, its headings and blocks are present and ordered, its extracted `make` targets are
members of the Makefile's phony list, and its extracted paths resolve in the working tree. None of that
needs a running container, and none of it is weakened by the restriction — the behavioural claims about
the rig are Plans 04-02's and 04-04's, and both already carry the 17/17 guard and the full-suite gate.

The alternative — serialising this plan into wave 3 — was rejected because a static-only verify leaves
nothing genuinely unproven here, and the parallelism this wave was designed for is worth keeping.

**Structure.**

*Opening.* One short paragraph saying what the demo shows and roughly how long it takes. Then the
pre-flight checklist (D-56), as a checkbox list, every item actionable: the hosts entry present (with

terminal — so it never appears in the test suite and always appears on the projector, directly under
the failure the room is meant to be reading.

*The beats, numbered, in the D-55 order:*

1. **Show OLD.** The proxied request landing on the old server, URL unchanged.
2. **The redirect contrast.** The other port returning a redirect and the client visibly ending on a
   different URL. This is the conceptual setup for the flip and belongs before it.
3. **Prime the SSH trust.** Connect in presenter mode while the selector is still on old. This step is
   load-bearing and must say so: the client's trust record is keyed on the name typed, so a connection
   made to a backend's own service name records a different, useless entry and the gotcha will not
   fire later. The target hard-codes the proxied hostname so this cannot be fumbled.
4. **The flip.** One word, one reload, and the identical request now answered by the new server —
   with the log tail and the status page as the evidence surfaces.
5. **The gotcha.** ⚠ Mark the heading destructive and name the command that re-arms it for the next
   take. The expected-output block is the transcribed warning: thirteen lines, a non-zero exit status
   stated explicitly, ending on the verification-failed line, and naming the client's trust record
   file with the offending line number. Show the fingerprint line as varying, not as a literal. Add a
   short "if it does not fire" note giving the three preconditions and the one caveat that an already
   open session keeps talking to its original backend for its whole life, so a terminal left open
   before the flip will show nothing.
6. **The wrong fix.** The client-side record deletion, shown working, transcribed verbatim, with no
   editorial in the expected-output block. The takeaway block carries the contrast: it took two steps
   rather than one, the second was trusting whatever answered next without checking, and it fixed
   exactly one machine — then the question to the room about how many machines hold that record.
7. **The right fix.** `make fix-hostkeys`, its transcript, and then the identical presenter command
   succeeding with the new server's banner. The takeaway is the phase's best single sentence:
   cryptographic identity inherited, application identity new. Point deliberately at the transferred
   public key still carrying its origin host's comment field — cosmetic, no protocol effect, and the
   most vivid available evidence that the new server is genuinely wearing the old server's identity.
   State that the fix survives a container restart, so the presenter cannot be ambushed mid-demo.
8. **Reset for the next take.** `make reset` as the documented headline path, with its measured
   duration; the fast in-place re-arm noted as the between-takes alternative with its measured
   duration. Say which one leaves which state.

**Every numbered step has exactly three blocks, under the step heading, always in this order:**
the command to run, the output to expect, and what to say. Nothing between them, and nothing after the

find the block they need by position rather than by reading, and Plan 04-04 asserts it mechanically.

failing. Frame that last one as a feature, in one line: the verify script answers "did the routing
land?", not "does the client trust what it landed on?" It is the sharpest available illustration of
why two connection modes exist.

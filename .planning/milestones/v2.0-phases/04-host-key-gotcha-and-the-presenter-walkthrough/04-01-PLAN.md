<objective>
Make the failure reachable on demand, and make the real fix one command.

Nothing in this plan creates the host-key mismatch — Phase 1 did that, deliberately, by generating
host keys at container start rather than in a build layer. What is missing is a way for the presenter
to *reach* the failure (a connection mode that remembers trust), a way to *fix* it the way a real
migration does (give the new server the old server's identity), and a way to *put it back* between
takes without a sixteen-second rebuild.

Purpose: KEY-01 through KEY-04. After this plan the presenter can run the entire host-key narrative by
hand — prime on OLD, flip, watch it break, fix it, watch it work — with no document and no test suite
involved. Plans 02, 03 and 04 assert it, script it and sign it off.

Output: `scripts/fix-hostkeys.sh`, `scripts/rearm.sh`, and three new `make` targets — `ssh`,
`fix-hostkeys`, `rearm`.

## Phase Goal

**As a** presenter, **I want to** deliberately trigger the SSH host-key failure after the cutover and
fix it live without touching the client, **so that** the audience sees the thing that actually breaks
on migration day and the answer that scales to a real fleet.

*(Direct restatement of the ROADMAP Phase 4 `**Goal:**` line — role, capability and outcome are all
present in that sentence; nothing has been invented.)*

**This plan's slice of that goal:** the mechanism. The narrative made runnable at the command line,
before any document describes it.

## Locked decisions this plan implements

| Decision | How this plan honours it |
|---|---|
| **D-47** the mismatch is already the default state; this phase makes the failure reachable, not real | Nothing here generates a mismatch that did not already exist. `rearm` *restores* the default state after the fix has removed it |
| **D-48 (as corrected by research)** no new mechanism holds the client's trust record | The presenter-mode target names no trust-record path at all. `compose.yaml` is not touched. | **D-49** the fix transfers `server-old`'s host keys to `server-new`, not an edit on the client | `scripts/fix-hostkeys.sh` reaches into the two backends and never into the client |
| **D-50** the fix is one memorable `make` target; the underlying commands stay visible | `make fix-hostkeys`, recipe body in a script, and the script prints each step as it runs so the audience sees what happened |
| **D-51** the fix is reversible so the gotcha can be re-armed | Two paths ship: `make reset` stays the documented headline, `make rearm` is the between-takes fast path. Both are recorded in this plan's decisions below |
| **D-52** two explicit named connection modes | `make ssh` is presenter mode. Test mode is Phase 3's existing pins and is not touched anywhere in this plan |
| **D-58** presenter mode pins the client's host-key self-update behaviour off | Without it the client rewrites its own trust record on the first successful post-fix connection and KEY-04's claim becomes unassertable |
| **D-59** copying the key files is not the fix | The daemon signal is inside the same target as the transfer, and the script fails loudly rather than reporting success if the presented key does not change |

## Plan-level decisions (Claude's discretion, exercised here)

- **Presenter mode is an option set on a `make` target, not an env var and not a config file.**
  Command-line options outrank the client's own ssh config, the target is one word on stage, and an
  option set carries its justification in the comment directly above it where a presenter and an
  audience can both read it.
- **Both re-arm paths ship, and `make reset` stays the documented headline.** This honours D-51
  verbatim while adding what a presenter doing three takes in a row actually wants: research measured
  the full rebuild at 16.5 s and the in-place re-arm at about 1 s. The suite needs a non-rebuilding
  re-arm regardless, because it cannot call a full teardown from inside itself.
- **The re-arm is in-place, never a container recreate.** nginx resolves its upstream addresses when
  it parses its configuration and this rig deliberately declares no runtime resolver, so recreating a
  backend can strand the proxy on an address that no longer answers. Research saw the address happen
  to be reused, which is the worst kind of hazard: it works in rehearsal and fails on stage.
- **The key files move as a stream between the two containers, never through a host path.** The
  project prohibits key material on the host filesystem, and a stream carries mode and ownership
  without a follow-up permission fix.
- **Both destructive targets take no arguments.** A "which server" parameter on a command whose first
  action is deleting host keys is a foot-gun with no upside; the demo has exactly one target.
- **Compose's trailing hint block is suppressed inside the presenter target.** It is printed after any
  non-zero exec but only when output is a terminal — so it never appears in the test suite and always
  appears on the projector, directly underneath the failure the audience is meant to be reading.

<assumption_delta_decision>
**Noun now primary:** *connection mode*. Phase 3 had one unnamed SSH option set; this phase introduces
a second, and the two have opposite intentions about trust.

**Decision: `add-alongside`, deliberately — not `promote`.** Unifying the two behind one general
option set is the exact move research measured as destructive: test mode discards trust so that 186
routing assertions cannot trip over a host-key change, and presenter mode remembers trust so that the
change is the only thing you see. A single generalised mode would convert every routing assertion in
the suite into a host-key assertion the moment the gotcha is armed.

**Accepted debt, and what would force a promote:** the two option sets are maintained in two places
(`section_ssh` plus `scripts/verify.sh` for test mode; the `ssh` target for presenter mode). A third
consumer of either mode would force a promote to a single named definition per mode. Two consumers of
one mode and one of the other does not justify the indirection, and the comments at each site name the
other mode explicitly so neither can be edited in ignorance of the other.
</assumption_delta_decision>

Create `scripts/fix-hostkeys.sh`, POSIX `sh`, following `scripts/flip.sh`'s conventions exactly:
deliberately not `set -e`, a `usage()` that exits 2, a header comment block that gives each step its
measured reason, and every container call through `docker compose exec -T`.

The script takes **no arguments** and is hard-wired to move keys from `server-old` into `server-new`
(threat T-04-01). Any argument at all is a usage error.

Sequence:

1. **Gate.** Refuse to run unless both backends are up, in the same shape `flip.sh` gates the flip.
   Exit non-zero with a message naming which one is missing.
2. **Announce.** Print a line naming what is about to happen, in the register the rest of the repo
   uses — this runs on a projector and the audience reads it.
3. **Transfer.** Stream the six host-key files out of `server-old`'s `/etc/ssh` and into
   `server-new`'s, using `tar` on both ends over two `docker compose exec -T` calls joined by a pipe.
   Six files, not three: the daemon offers ed25519, rsa and ecdsa, and a client that has recorded
   more than one of them objects to whichever gets negotiated. `tar` carries mode and ownership, so
   no permission fix follows. Note in the comment that this particular pipe is safe and why: the
   Phase 3 prohibition is specifically on placing an `ssh` invocation on the left of a pipe, because
   a pipeline reports only the last command's status; here the second `tar`'s status is exactly the
   one wanted.
4. **Signal.** Send a hangup to the running daemon on `server-new`, reading its process id from the
   pid file the supervisor's program writes. This step is the fix (D-59): the daemon loads host keys
   once at startup and holds them in memory, and research measured a connection made after a verified
   successful transfer still failing on the old in-memory key. A hangup makes it re-exec, preserving
   its process id so the supervisor never notices. Do not restart the container and do not ask the
   supervisor to restart the program: a recreate regenerates the very keys just installed and can
   move the container's address out from under nginx.
5. **Prove.** Read the ed25519 fingerprint from each backend and compare them. Print the fingerprint.
   Exit non-zero with an explicit message if they differ — the script must not be able to report
   success on a transfer that did not land.
6. **Report.** Print a short done line. Do not touch the client container at all: this fix reaches
   into the two servers only, which is the whole of D-49.

Also record in the header comment, for the reader who arrives at this file cold:
- that the transferred public key files keep their original comment field naming their origin host,
  that this is cosmetic with no protocol effect, and that it must be narrated rather than "fixed" —
  it is the most vivid available evidence that the new server is wearing the old server's identity;
- that the effect of this script survives a container restart, because the gap-filling key generator
  in the entrypoint only creates keys that are missing;
- that `scripts/rearm.sh` is the reverse of this script.
  </action>

2. **Announce.** One line naming what is about to happen.

   the failure mode this step exists to catch. Print both fingerprints.

  <action>
Add three targets to `Makefile` and extend `.PHONY` to name all three. Keep Make 3.81 discipline: tab
indented recipes, no 4.x-era directives, doubled dollar signs where a shell variable is meant.

**`ssh` — presenter mode.** One `docker compose exec client ssh` invocation against `demo@` the demo
hostname, with the target hard-coded so it cannot be fumbled. The trust record is keyed on the name
the client typed, so a connection made to a backend's own service name records a different, useless
entry and the gotcha will not fire; hard-coding the proxied hostname removes that failure entirely.

Two options and one environment variable, each with its reason in the comment above the target:
- `StrictHostKeyChecking=accept-new` — records an unseen host silently, so the priming beat has no
  prompt and no dead air, and still refuses a **changed** key with the full warning. The default
  setting prompts and blocks in a terminal, and with no terminal exits non-zero with a bare
  verification-failed line; research measured one probe hanging for a full two-minute timeout.
- `UpdateHostKeys=no` — without it the client rewrites its own trust record on the first successful
  post-fix connection, appending the server's other host keys and leaving a backup file. Research
  measured 95 bytes becoming 837. With it, the record is provably unchanged across the whole
  narrative and KEY-04's "no client-side edit" becomes a checksum comparison rather than a claim
  about intent.
- `DOCKER_CLI_HINTS=false` in the recipe's environment — Compose prints a trailing hint block after
  any non-zero exec, but only when output is a terminal, so it never appears in the suite and always
  appears on the projector, directly under the failure the room is meant to be reading.

The comment must state plainly that this target is **presenter mode**, that Phase 3's test mode is
its complement and lives in `section_ssh` and `scripts/verify.sh`, and that `accept-new` is not the
same thing as switching host-key checking off. Presenter mode must carry neither of Phase 3's
test-mode pins and must name no trust-record path at all — the default location in the client
container's writable layer is what couples the client's trust lifetime to the backends' key lifetime,
and naming a path is the first step toward a persistence mechanism that would make the gotcha fire
before the flip.

**`fix-hostkeys`** and **`rearm`** — thin wrappers over `scripts/fix-hostkeys.sh` and
`scripts/rearm.sh` respectively, in the same shape as `flip` and `verify`. Comment `rearm` as the
